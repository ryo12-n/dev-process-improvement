## 2. 機能要件

### FR-001: 書籍の CRUD 管理

- **説明**: 書籍情報の登録・参照・更新・削除を行う API エンドポイントを提供する。登録時には ISBN を指定でき、外部 API（OpenBD 等）から書誌情報を自動取得する機能を含む。
- **優先度**: high
- **ユーザーストーリー**:
  - 管理者として、新しい書籍を ISBN で登録したい。なぜなら手入力を減らし正確な書誌情報を取り込みたいからだ。
- **受入基準**:
  - [ ] 書籍を ISBN 指定で登録でき、書誌情報（タイトル・著者・出版社・出版日）が自動取得されること
  - [ ] 書籍の一覧取得・個別取得・更新・削除が API 経由で行えること
  - [ ] ISBN が無効または外部 API で見つからない場合、手動入力にフォールバックできること

### FR-002: 書籍のタグ・カテゴリ管理

- **説明**: 書籍にタグやカテゴリを付与し、分類・整理できる機能を提供する。
- **優先度**: medium
- **ユーザーストーリー**:
  - （該当なし）
- **受入基準**:
  - [ ] 書籍に複数のタグを付与・削除できること
  - [ ] 書籍にカテゴリを設定できること
  - [ ] タグ・カテゴリの CRUD 操作が行えること

### FR-003: 書籍検索

- **説明**: タイトル・著者・ISBN・タグをキーワードとして書籍を検索できる機能を提供する。
- **優先度**: high
- **ユーザーストーリー**:
  - 利用者として、タイトル・著者・タグでキーワード検索したい。なぜなら目的の本を素早く見つけたいからだ。
- **受入基準**:
  - [ ] タイトル・著者・ISBN・タグのいずれかで部分一致検索ができること
  - [ ] 複数条件の AND 検索ができること
  - [ ] 検索結果がページネーション付きで返されること
  - [ ] 1000冊規模で検索レスポンスが 500ms 以内であること

### FR-004: 貸出リクエスト・承認ワークフロー

- **説明**: 利用者が書籍の貸出リクエストを送信し、管理者が承認・却下する一連のワークフローを提供する。
- **優先度**: high
- **ユーザーストーリー**:
  - 利用者として、書籍の貸出リクエストを送信したい。なぜなら口頭やりとりをなくし記録を残したいからだ。
  - 管理者として、貸出リクエストを承認・却下したい。なぜなら在庫状況を見て判断したいからだ。
- **受入基準**:
  - [ ] 利用者が書籍を指定して貸出リクエストを送信できること
  - [ ] 管理者が貸出リクエストの一覧を確認できること
  - [ ] 管理者が貸出リクエストを承認または却下できること
  - [ ] 承認時に貸出レコードが作成され、返却期限が設定されること
  - [ ] 既に貸出中の書籍にはリクエストできないこと（またはキューに入ること）

### FR-005: 返却・貸出履歴管理

- **説明**: 貸出中の書籍の返却処理と、過去の貸出履歴の記録・参照機能を提供する。
- **優先度**: high
- **ユーザーストーリー**:
  - 利用者として、自分の貸出中の書籍一覧を確認したい。なぜなら返却期限を把握したいからだ。
- **受入基準**:
  - [ ] 利用者が自分の貸出中書籍一覧を取得できること
  - [ ] 貸出中の書籍を返却済みに更新できること
  - [ ] 過去の貸出履歴（貸出日・返却日・利用者）を参照できること

### FR-006: 延滞アラート

- **説明**: 返却期限を超過した貸出の一覧を取得する機能を提供する。
- **優先度**: medium
- **ユーザーストーリー**:
  - 管理者として、延滞中の貸出一覧を確認したい。なぜなら督促を効率化したいからだ。
- **受入基準**:
  - [ ] 返却期限超過の貸出一覧を API で取得できること
  - [ ] 延滞日数が計算されて返されること
  - [ ] 利用者別・書籍別でフィルタリングできること

### FR-007: ユーザー管理・認証認可

- **説明**: 管理者と一般利用者のロールを持つユーザー管理機能と、JWT ベースの認証・認可を提供する。
- **優先度**: high
- **ユーザーストーリー**:
  - （該当なし）
- **受入基準**:
  - [ ] ユーザー登録・ログイン・ログアウトが行えること
  - [ ] JWT トークンによる認証が実装されていること
  - [ ] 管理者ロールと利用者ロールで API のアクセス制御が行われること
  - [ ] パスワードが bcrypt でハッシュ化されて保存されること

## 3. 非機能要件

### NFR-001: 検索レスポンス性能

- **カテゴリ**: performance
- **説明**: 1000冊規模のデータに対する検索クエリが許容時間内に完了すること。
- **メトリクス**: レスポンスタイム
- **目標値**: 500ms 以内

### NFR-002: 認証・認可セキュリティ

- **カテゴリ**: security
- **説明**: JWT ベースの認証・認可を実装し、パスワードは bcrypt でハッシュ化する。
- **メトリクス**: 認証方式
- **目標値**: JWT + bcrypt ハッシュ

### NFR-003: システム可用性

- **カテゴリ**: availability
- **説明**: 社内ツールとして必要十分な可用性を確保する。
- **メトリクス**: 稼働率
- **目標値**: 99.5%

### NFR-004: 同時アクセス・データ規模

- **カテゴリ**: scalability
- **説明**: 想定される利用規模に対応できるスケーラビリティを確保する。
- **メトリクス**: 同時ユーザー数 / 蔵書数上限
- **目標値**: 同時 50 ユーザー / 蔵書 10,000 冊

## 4. 制約事項

- 言語/フレームワーク: TypeScript + Express.js
- データベース: PostgreSQL 15+
- インフラ: Docker Compose でローカル開発、将来的に AWS ECS デプロイを想定
- API 仕様: OpenAPI 3.0 でドキュメント化
- CI: GitHub Actions を使用

## 5. 前提条件

- 社内チーム向けの利用を前提とする（パブリック公開は想定しない）
- チーム内の技術スタックが TypeScript + Express.js である
- 蔵書規模は 10,000 冊以内を当面の上限とする

## 6. スコープ外

- フロントエンド UI（API のみ提供。UI は別プロジェクトで対応）
- 電子書籍ファイルの保管・配信
- 外部図書館システムとの連携
- 多言語対応（日本語のみ）
- モバイルアプリ
