# プロジェクトルール: dev-process-improvement

## 概要
このリポジトリは「開発プロセス改善」を継続的に回すためのもの。
改善施策ごとに `sessions/initiatives/<施策名>/` フォルダを作り、L1・L2セッションで分担して進める。

## セッション構成

| セッション | ロール | 責務 |
|-----------|--------|------|
| L1 (マネージャー) | 調査・意思決定・計画・ゲート判定 | sessions/initiatives/ 内の proposal, plan, tasks, gate_review を管理 |
| L2 (ワーカー/実施) | タスク実行・作業記録・レポート・課題起票 | sessions/initiatives/ 内の work_log, work_report, issues を管理 |
| L2 (ワーカー/評価) | 評価計画・評価実施・レポート・課題起票 | sessions/initiatives/ 内の eval_plan, eval_report, issues を管理 |

## ファイル命名規則

- sessions/initiatives/ 内のファイルは番号付きプレフィックス（00〜08）で管理する
- 作業履歴（work_log）はタイムスタンプ付きで追記する
- 課題（issues）は連番IDを振る

## 施策管理の必須化

**すべてのタスクは、大小に関わらず initiative として管理する。**

- バックログから施策を開始する場合は L1-manager フローに従い、`sessions/initiatives/<施策名>/` ディレクトリを作成してから作業に着手する
- initiative ディレクトリなしでの作業着手は禁止
- 小規模タスクでも最低限 00_proposal.md, 01_plan.md, 02_tasks.md を作成する
- 事後的な initiative 作成は手戻りが発生するため、必ず事前に作成すること

## 禁止事項

- L2セッションが plan や gate_review を直接変更すること（L1の責務）
- L1セッションが work_log を直接編集すること（L2の責務）
- 施策フォルダの外にあるファイルへの無断変更
- フェーズゲートを通過していない施策の次フェーズ着手
- initiative ディレクトリを作成せずに作業を開始すること

## コマンド

- 新規施策開始: 「○○の改善施策を開始して」→ _template/ をコピーして sessions/initiatives/<施策名>/ を作成
- 状況確認: 「施策一覧を見せて」→ sessions/initiatives/ 内の全施策のステータスを一覧表示
- ゲート判定: 「○○のフェーズゲート判定をして」→ gate_review を更新

## ディレクトリ構成と用途

| ディレクトリ | 用途 |
|-------------|------|
| `inbox/` | 未判断のアイデア・提案・気づき（`.md` 1ファイル単位）。トリアージで処理される |
| `refs/` | 他PJの実装・構成など、施策の参考にする参照物置き場。inbox にポインターメモを置き、本体はここに置く。外部リポジトリ固有の知見は `<リポ名>/knowledge.md` に蓄積する（`manager-common-policy` §5.3 参照）。**参照物格納時は用途・有効期限のメタ情報を必ず記載すること** |
| `backlog/` | トリアージ承認済みの施策候補。ルート直下に CSV・README・`_template.md`、`entries/` に施策 `.md` ファイルを格納 |
| `sessions/initiatives/` | 進行中・完了済みの施策 |
| `sessions/triage/` | トリアージセッションの作業フォルダ |
| `sessions/metacognition/` | メタ認知セッションの作業フォルダ（施策アーカイブの横断分析・振り返り） |
| `issues/` | 判明した課題・技術的制約の詳細管理。ルート直下に README・`_template.md`、`entries/` に `ISS-XXX.md` を格納。CSV（全体俯瞰ハブ）と対になる |
| `docs/` | ルールの人間向け可視化・ログ |
| `templates/` | 施策テンプレート |

### inbox と refs の使い分け

- **アイデア・提案**（判断待ち）→ `inbox/<題名>.md` に直接書く
- **参照物**（他PJの実装など実体ファイルを持ち込む場合）→ `refs/<名前>/` に本体を置き、`inbox/ref_<名前>.md` にポインターメモを置く

### inbox と issues の使い分け

| 分類 | 性質 | 置き場 | 例 |
|------|------|--------|-----|
| 判明した技術的制約・バグ | 事実・追跡対象 | `issues/` + CSV | 「deny リストが SDK 環境で機能しない」 |
| ルール改善の種・次施策候補・気づき | アイデア・未判断 | `inbox/` → トリアージ | 「改善提案に環境を明記すべき」 |

- **issues/**: 「今起きている問題」「判明した事実」を記録し、CSV で追跡する
- **inbox/**: 「今後やるかもしれないこと」「気づき」を記録し、トリアージで判断を待つ

## ドキュメント階層方針

- **README はエントリーポイントに徹する**: README には概要・クイックスタートのみ記載し、150行以内に保つ。詳細は `docs/` に分離する
- **3層構造を基本とする**: `README.md`（概要） → `docs/<ガイド>.md`（中間説明） → `docs/design/`（詳細設計）の順で抽象度を段階的に深める
- **150行を超える場合は分離を検討する**: README に限らず、単一ドキュメントが肥大化した場合は詳細を `docs/` に切り出す

## ドキュメント整合性ルール

`.claude/skills/` 配下のスキル・エージェント定義が正の情報源（Source of Truth）。`docs/workflow.md` はその人間向け可視化。

**ルールファイルを変更した場合は、必ず `docs/workflow.md` も合わせて更新すること。**

### docs/ 文書分類

`docs/` 配下の文書は以下の3カテゴリに分類し、カテゴリごとに管理ルールを適用する。

#### カテゴリ定義

| カテゴリ | 定義 | 整合性チェック義務 |
|---------|------|------------------|
| 可視化文書 | `.claude/skills/` や `.claude/rules/` の人間向け可視化。正の情報源はルールファイル側 | あり（ルール変更時に連動更新必須） |
| 運用ガイド | 独立した情報源として機能する文書。ルールファイルとの1対1対応を持たない | なし（独立管理） |
| 参考資料（ドラフト） | 将来 `.claude/skills/` への統合を予定する文書。凍結扱い | なし（ステータスバナー必須） |

#### 現在の docs/ ファイル分類

| ファイル | カテゴリ | 備考 |
|---------|---------|------|
| `docs/workflow.md` | 可視化文書 | `.claude/skills/` 全体の人間向け可視化 |
| `docs/management-system-guide.md` | 運用ガイド | 課題管理システムの運用手順 |
| `docs/subagent-permission-guide.md` | 運用ガイド | サブエージェント権限の運用手順 |
| `docs/implementation.md` | 運用ガイド | 実装ガイド |
| `docs/coordination-protocol-guideline.md` | 参考資料（ドラフト） | 協調プロトコルのガイドライン。一部統合済み |
| `docs/git-worktree-guideline.md` | 参考資料（ドラフト） | git worktree のガイドライン。一部統合済み |

#### 管理ルール

- **可視化文書**: ルールファイル変更時に連動更新すること（既存の整合性ルールに従う）
- **運用ガイド**: 独立した情報源として管理する。ルールファイルとの整合性チェックは不要
- **参考資料（ドラフト）**: 以下のステータスバナーをファイル冒頭に付与し、文書の位置づけを明示する。Skills 統合が完了した文書は削除を検討する

```markdown
> **文書ステータス**: ドラフト（参考資料）
> **正の情報源への統合**: 未着手 / 一部統合済み / 統合完了
> Skills 統合後に削除を検討する。
```

- **新規 docs/ ファイル作成時**: 上記3カテゴリのいずれかを明記し、本テーブルに追加すること

## セッション終了時の気づき記録

協働スタイルや設計に関する気づきが生まれたセッションの終了時に、以下の手順で記録する。

1. **AIが要約を提示する** — そのセッションで生まれた気づきを箇条書きでまとめる
2. **人間が承認・修正する** — 不要なものを削除、表現を調整
3. **`inbox/` に `.md` ファイルとして記録する** — 1件1ファイル形式で、トリアージで処理される

### 記録の対象

- ルール・workflow の改善につながる気づき
- 協働スタイル（確認の粒度・委譲のやり方など）の変化
- 次のトリアージで判断してほしい問い

### 記録しないもの

- 単発の技術的なQ&A
- その場で完結したタスクの細部
- 感想・褒め言葉（気づきに昇華できないもの）

### 蒸留後の扱い

トリアージでルール・docsに反映されたエントリは `inbox/` から削除する。git履歴が証跡となる。

## 文書分類ポリシー

ドキュメントの読者と読み込み頻度に応じて適切な場所に配置する。ルートの CLAUDE.md に定義されたポリシーに準じる。

| 分類 | 読者 | 配置先 | 用途例 |
|------|------|--------|--------|
| Claude 常時参照 | Claude | `.claude/rules/` | コミットメッセージ規約等（全セッション共通） |
| Claude オンデマンド参照（スキル） | Claude | `.claude/skills/` | セッションルール（l1-manager, triage-manager 等） |
| Claude 独立コンテキスト（エージェント） | Claude | `.claude/skills/*/agents/` | ワーカー・評価者エージェント（l2-worker, triage-worker 等） |
| 人間用 | 人間 | `docs/`, `README.md` | workflow.md、運用手順 |

**判断基準**: 「毎回のリクエストで参照が必要か？」→ Yes なら `rules/`、No なら `skills/`。

## 環境分離ポリシー

`backlog.csv` の `対象リポジトリ` 列で施策の適用先を管理する。

| 対象リポジトリ | 説明 | 実行環境 |
|--------------|------|---------|
| `dev-process-improvement` | プロセス改善リポジトリ内で完結する施策 | CLI / SDK いずれでも可 |
| `ai-driven-dev-patterns` | ai-driven-dev-patterns リポジトリへの変更を含む施策 | ローカル CLI 推奨（`--add-dir` でコンテキスト参照） |
| `横断` | 両リポジトリにまたがる施策、または環境共通の施策 | ローカル CLI 推奨 |

### 運用ガイドライン

- **施策起票時**: `backlog/entries/<施策名>.md` に対象リポジトリを明記する。`backlog.csv` の `対象リポジトリ` 列も同時に記入する
- **環境依存の制約**: Claude Code Web（SDK 環境）は1セッション=1リポジトリの制約がある（ISS-027）。`ai-driven-dev-patterns` への変更が必要な施策はローカル CLI 環境での実施を推奨する
- **改善提案を含むタスク**: `02_tasks.md` の完了条件に「適用対象環境（CLI / SDK / 横断）を明記すること」を含める（l1-manager のタスクリスト作成ガイドに準拠）

## リポジトリ間の関係

本リポジトリ（dev-process-improvement）は ai-driven-dev-patterns のコンテキストに **依存する側** である。

- ai-driven-dev-patterns のリソース（roles/、CLAUDE.md、openspec/ 等）を `claude --add-dir` 等で参照し、改善結果を書き戻す
- 課題管理: `プロセス改善_課題管理.csv` で独立管理（ai-driven-dev-patterns の `課題管理.csv` とは別体系）
