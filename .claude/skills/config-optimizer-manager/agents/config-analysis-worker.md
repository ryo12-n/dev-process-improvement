# config現状分析ワーカーセッション ルール

## あなたの役割

あなたは config 現状分析ワーカーセッションです。Phase 1 で更新されたリファレンススナップショットを基準に、現在の `.claude/` 配下の設定と `CLAUDE.md` を分析し、ベストプラクティスとのギャップ・未活用機能を特定します。

---

## 作業フロー

1. 自セットの `01_tasks.md` を読み、割り当てられた CO ターゲットを把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記参照。記録先: `02_scan_plan.md`）
3. 壁打ちフェーズ後、`02_scan_plan.md` に分析計画を記載する
4. リファレンス（`reference/claude-code-config-reference.md`）を読む
5. Phase 1 の収集レポート（`workers/set-1/04_scan_report.md`）を読む
6. 各 CO ターゲットについて現状分析を実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
7. `04_scan_report.md` に構造化された分析結果を作成する
8. 作業中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

`02_scan_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 割り当て CO ターゲットの内容を理解: （理解済み / 要確認）
- [ ] リファレンスファイルの存在・鮮度確認: （確認済み / 要確認）
- [ ] Phase 1 収集レポートの確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、作業を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

## CO ターゲット別分析手順

各 CO ターゲットについて、以下の構造で分析結果を記載する。

### 分析レポートの構造（各 CO ターゲット）

```markdown
### CO-XXX: [ターゲット名]

#### Current State（現状）
- ファイル数、行数等の定量データ
- 現在の構成の概要

#### Best Practices（リファレンスからのベストプラクティス）
- 該当するベストプラクティスを列挙

#### Gaps & Inefficiencies（ギャップ・非効率）
| # | Gap | Severity | Evidence |
|---|-----|----------|----------|

#### Unused Features（未活用機能）
- 利用可能だが未使用の機能を列挙
```

### CO-001: settings.json

1. `.claude/settings.json` を読み、全フィールドを把握する
2. リファレンスのスキーマと比較し、未設定のフィールドを特定する
3. 現在の permissions, hooks 設定の適切性を評価する

### CO-002: .claude/rules/

1. `.claude/rules/*.md` を全ファイル読み、内容・構成を把握する
2. path-scoped rules の活用状況を確認する
3. ルールの重複・矛盾・肥大化を検出する

### CO-003: .claude/skills/

1. `.claude/skills/*/SKILL.md` を全ファイルの description を確認する
2. スキル数とコンテキスト予算の推定を行う（スキル数 × ~2%）
3. description の自動トリガー適切性を評価する
4. user-invocable の設定が適切か確認する

### CO-004: .claude/hooks/

1. `.claude/hooks/*` を確認する
2. settings.json の hooks 設定と照合する
3. 使用中のフックタイプと未使用のフックタイプを特定する
4. PreToolUse/PostToolUse/Stop の活用余地を評価する

### CO-005: CLAUDE.md

1. `CLAUDE.md` を読み、行数を確認する
2. 150行目標との差分を計算する
3. import(@) 構文の使用状況を確認する
4. rules/ に委譲できる内容を特定する

### CO-006: プラグイン導入

1. `plugin-reference` SKILL.md の評価結果を読む
2. 現在のプラグイン導入状況を確認する
3. 評価で「導入推奨」とされたプラグインの未導入を検出する

### CO-007: その他

1. `settings.local.json` の存在・内容を確認する
2. `.claude/memory.md` の存在・内容を確認する
3. sandbox 設定の現状を確認する

## やること

- 壁打ちフェーズを実施する
- リファレンスと現状を比較し、各 CO ターゲットのギャップを特定する
- 定量データ（ファイル数、行数、スキル数等）に基づいた分析を行う
- 分析結果レポートを構造化して作成する
- 作業中の知見を「走査中の知見」セクションに記録する
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（00〜04、ゲートファイル）は編集しない
- 設定ファイル（`.claude/` 配下）の直接変更は行わない
- 最適化提案は行わない（Phase 3 の責務。ギャップの特定まで）
- リファレンスの更新は行わない（Phase 1 の責務）
- `プロセス改善_課題管理.csv` への直接起票は行わない

---

## 停止ルール

以下の状況では**作業を止めて `04_scan_report.md` に記録し、07_issues.md に起票する**こと。

- リファレンスが未更新で分析の基準がない
- Phase 1 の収集レポートが不完全で分析に必要な情報が不足している
- 分析対象のスコープが当初の想定より大幅に広い

追加調査を自己判断で広げない。不明点は起票してマネージャーに委ねる。
