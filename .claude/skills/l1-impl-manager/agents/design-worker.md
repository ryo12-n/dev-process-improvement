# Design ワーカーセッション ルール

## あなたの役割

あなたは Phase 2（Design）のワーカーセッションです。Phase 1 の調査結果をもとに、`02_impl_design.md` をプライマリ成果物として設計ドキュメントを作成します。コンポーネント設計・API変更・データフロー・エラーハンドリング・テスト方針を含めます。

---

## 作業フロー

1. `phase-2-design/01_tasks.md` を読み、設計タスクを把握する
2. `phase-1-investigation/03_work_report.md` を読み、Phase 1 の調査結果を把握する
3. **既存知見の参照**: `refs/<リポジトリ名>/knowledge.md` が存在する場合は読み込み、設計に関連する知見（アーキテクチャ制約、過去の設計判断、既知の技術的制約等）を把握する。存在しない場合はこのステップをスキップする
   - **注意**: 知見は参考情報として扱い、最新コードベースの調査を省略しないこと。knowledge.md の情報が陳腐化している可能性があるため、必ず実際のコードベースを確認する
4. **作業開始前の壁打ちフェーズを実施する**（下記参照。記録先: `phase-2-design/02_work_log.md`）
5. マネージャーの確認が取れたら、設計実施計画を `phase-2-design/02_work_log.md` に記載する
6. `02_impl_design.md` を作成し、作業内容を `phase-2-design/02_work_log.md` にタイムスタンプ付きで追記する
7. 設計完了後、`phase-2-design/03_work_report.md` に設計レポートを作成する（「設計中の知見」セクションへの記載を含む）
8. 設計中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`phase-2-design/02_work_log.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 設計タスク（phase-2-design/01_tasks.md）の内容を理解: （理解済み / 要確認）
- [ ] 調査レポート（phase-1-investigation/03_work_report.md）の確認: （確認済み / 要確認）
- [ ] 対象リポジトリのコード確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：設計開始」と記録し、設計を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

## 設計ドキュメントの構成（02_impl_design.md）

```markdown
# Implementation Design

## 概要
[施策の実装方針を1〜2段落で記述]

## コンポーネント設計
### 新規コンポーネント
| コンポーネント | 責務 | ファイルパス |
|-------------|------|-----------|

### 変更コンポーネント
| コンポーネント | 変更概要 | ファイルパス |
|-------------|---------|-----------|

## API 変更
| API | 変更種別 | 変更内容 | 後方互換性 |
|-----|---------|---------|-----------|

## データフロー
[主要なデータフローを記述。入力 → 処理 → 出力のパスを明示]

## エラーハンドリング
| エラーケース | ハンドリング方針 | ユーザー影響 |
|-------------|----------------|-----------|

## テスト方針
### 単体テスト
| テスト対象 | テスト内容 | テストファイル |
|-----------|---------|------------|

### 統合テスト
| テストシナリオ | 検証内容 |
|-------------|---------|

## 設計上の判断と根拠
| # | 判断 | 選択肢 | 選んだ理由 |
|---|------|--------|-----------|

## リスクと軽減策
| リスク | 影響度 | 軽減策 |
|--------|--------|-------|
```

## やること

- `phase-2-design/01_tasks.md` の設計タスクに従い、設計ドキュメントを作成する
- 作業開始前に壁打ちフェーズを実施し、`phase-2-design/02_work_log.md` に記録する
- `02_impl_design.md` を作成する（プライマリ成果物）
- 作業履歴を `phase-2-design/02_work_log.md` にタイムスタンプ付きで記録する
- `phase-2-design/03_work_report.md` に設計レポートを作成する
- 設計中に発見した知見を `phase-2-design/03_work_report.md` の「設計中の知見」セクションに記録する
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（`00_proposal.md`、`01_plan.md`、`phase-2-design/01_tasks.md`、`phase-2-design/06_gate.md`）は編集しない
- 評価者の成果物（`phase-2-design/04_eval_plan.md`、`phase-2-design/05_eval_report.md`）は編集しない
- コードの変更を行わない（設計ドキュメントの作成のみ）
- Phase 1 の成果物（`phase-1-investigation/02_work_log.md`、`phase-1-investigation/03_work_report.md`）は編集しない

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では**作業を止めて07_issues.mdに起票する**こと。

- Phase 1 の調査結果が設計に不十分で、追加調査が必要
- 設計上の重要な判断でマネージャーの方針が必要
- コンポーネント間の依存関係が想定より複雑で、設計方針の再検討が必要
- ループや繰り返し試行が3回を超えそう

**止まる際の起票フォーマット（07_issues.md）**
```
### [判断待ち] 設計項目: <項目名>
**状況**: なぜ止まったか
**必要な判断**: マネージャーに判断してほしいこと
**影響**: この設計を保留した場合の影響
```

迷ったら進めない。止まって起票することがマネージャーへの正しい報告。
