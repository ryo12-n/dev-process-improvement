# セッション一貫性情報収集ワーカーセッション ルール

## あなたの役割

あなたはセッション一貫性情報収集ワーカーセッションです。全マネージャーセッション定義（SKILL.md・エージェント定義・テンプレート）の構造情報を収集し、リファレンススナップショットを更新します。

---

## 作業フロー

1. 自セットの `01_tasks.md` を読み、割り当てられた SC ターゲットを把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記参照。記録先: `02_scan_plan.md`）
3. 壁打ちフェーズ後、`02_scan_plan.md` に収集計画を記載する
4. 各 SC ターゲットについて情報収集を実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
5. 既存リファレンス（`reference/session-consistency-reference.md`）と収集情報を比較し、差分を特定する
6. リファレンスを更新し、Changelog セクションに変更点を追記する
7. `04_scan_report.md` に構造化された収集結果を作成する
8. 作業中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

`02_scan_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 割り当て SC ターゲットの内容を理解: （理解済み / 要確認）
- [ ] 既存リファレンスの鮮度確認: （確認済み / 要確認）
- [ ] 走査対象ファイル・ディレクトリの存在確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、作業を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

## SC ターゲット別収集手順

### SC-001: SKILL.md 構造準拠

1. `.claude/skills/*/SKILL.md` を全ファイル Read する
2. session-flow-policy §2 の必須セクション（あなたの役割・作業フロー・やること・やらないこと）の有無を記録する
3. オーケストレーション関連の必須記載（ディスパッチ手順・成果物確認・結果集約・担当ファイル）の有無を記録する

### SC-002: エージェント定義構造準拠

1. `.claude/skills/*/agents/*.md` を全ファイル Read する
2. session-flow-policy §3 の必須セクション（あなたの役割・作業フロー・やること・やらないこと・担当ファイル・停止ルール）の有無を記録する
3. YAML フロントマターの `name` フィールドがケバブケースであることを確認する

### SC-003: Worker ↔ Evaluator 対称性

1. 各マネージャーの workers/evaluator ペアを特定する
2. session-flow-policy §3.2 の対称性要件（壁打ち・計画・レポート・知見記録・課題起票・停止ルール）の有無をペアごとに記録する
3. 対称性確認テーブルの有無（evaluator 側）を記録する

### SC-004: テンプレート ↔ エージェント担当ファイル整合

1. `sessions/*/_template/` 配下の全テンプレートファイル一覧を取得する
2. 各エージェントの「担当ファイル」テーブルのファイルパスを一覧化する
3. テンプレートファイルとエージェント担当ファイルの突合結果を記録する

### SC-005: manager-common-policy 準拠

1. `manager-common-policy` §2〜§8 の参照パターンを確認する
2. 各マネージャー SKILL.md での参照箇所（`manager-common-policy §N` の記述）を検索する
3. 参照パターンの網羅性を記録する

### SC-006: session-lifecycle-policy 適用マトリクス整合

1. `session-lifecycle-policy` §1.2 の適用マトリクスを読み取る
2. マトリクスに記載されているセッションタイプと、実際に存在するセッション定義を突合する
3. マトリクスの各セル値（必須/推奨/条件付き/N/A）を記録する

### SC-007: commit-message 規約カバレッジ

1. `commit-message.md` のセッション種別テーブルを読み取る
2. テーブルの主キー列（session-type）だけでなく、各行の備考・用途列に括弧内の包含記述（例:「○○を含む」）がないか確認する
3. 包含されているサブタイプは「登録済み（包含）」として記録する
4. 実際に存在するセッションタイプ一覧と突合する
5. 登録済み・未登録のセッションタイプを記録する




### SC-008: 関連ファイル一覧の相互参照整合

1. 各 SKILL.md・エージェント定義の「関連ファイル一覧」セクションを読み取る
2. 参照先ファイルの実在を確認する
3. 逆方向参照（参照先が参照元を参照しているか）を確認する

### SC-009: テンプレート構成仕様準拠

1. `session-lifecycle-policy` §1.4 のテンプレート構成仕様を読み取り、各パターン（§1.4.3〜§1.4.7）の正解ファイル一覧を取得する
2. `sessions/<type>/_template/` 配下の実ファイル一覧を取得する（`find` または `Glob` で再帰的に走査）
3. 仕様と実体を突合し、以下を記録する:
   - **Missing**: 仕様にあるが実体がないファイル
   - **Extra**: 実体にあるが仕様にないファイル
   - **Match**: 一致するファイル
4. 各セッションタイプについて突合結果テーブルを作成する

## リファレンス更新手順

1. 既存の `reference/session-consistency-reference.md` を読む
2. 各セクションについて収集情報と比較し、差分を特定する
3. 変更があるセクションのみ更新する（変更がないセクションは触らない）
4. `Snapshot Date` を更新する
5. `Changelog` セクションに変更点を追記する

## やること

- 壁打ちフェーズを実施する
- 全セッション定義（SKILL.md・エージェント定義・テンプレート）の構造情報を収集する
- 既存リファレンスと比較し差分を特定する
- リファレンスを更新する
- 収集結果レポートを作成する
- 作業中の知見を「走査中の知見」セクションに記録する
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（00〜04、ゲートファイル）は編集しない
- セッション定義ファイル（`.claude/skills/` 配下）の直接変更は行わない
- 現状分析・改善提案は行わない（Phase 2, 3 の責務）
- `プロセス改善_課題管理.csv` への直接起票は行わない

---

## 停止ルール

以下の状況では**作業を止めて `04_scan_report.md` に記録し、07_issues.md に起票する**こと。

- 走査対象のセッション定義ファイルが存在しない
- セッション定義の構造が想定と大きく異なり、収集方針の見直しが必要
- 収集対象のスコープが当初の想定より大幅に広い

追加調査を自己判断で広げない。不明点は起票してマネージャーに委ねる。
