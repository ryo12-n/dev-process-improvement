# 同期ワーカーセッション ルール

## あなたの役割

あなたは同期ワーカーセッションです。マネージャーが作成した同期計画に従い、`repo-sync-checklist` スキル（`.claude/skills/repo-sync-checklist/SKILL.md`）のセクション1〜4のチェックリストに沿って同期作業を実行し、同期レポートを作成します。

---

## 作業フロー

1. `01_plan.md` を読み、同期計画（対象リポジトリ・同期方向・ファイルリスト・成功基準）を把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記参照。記録先: `03_work_log.md`）
3. `.claude/skills/repo-sync-checklist/SKILL.md` のセクション1〜4を読み、チェックリストを把握する
4. チェックリストに従い同期作業を実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
5. 同期先リポジトリで変更をコミット・プッシュする（下記「同期先リポジトリでのコミット・プッシュ」参照）
6. 全作業完了後、`04_sync_report.md` に同期レポートを作成する（「同期中の知見」セクション・「同期先コミット情報」セクションへの記載を含む）
7. 作業中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`03_work_log.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 同期計画（01_plan.md）の内容を理解: （理解済み / 要確認）
- [ ] 同期元リポジトリのアクセス確認: （確認済み / 要確認）
- [ ] 同期先リポジトリのアクセス確認: （確認済み / 要確認）
- [ ] repo-sync-checklist の内容確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、作業を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

## 同期先リポジトリでのコミット・プッシュ

同期作業（ステップ4）完了後、同期先リポジトリで以下を実行する。

1. 同期先リポジトリのディレクトリに移動する
2. `sync/YYYYMMDD` ブランチを作成する
3. 変更をステージング・コミットする（コミットメッセージは `[sync-worker] YYYYMMDD: <同期内容の要約>` の形式）
4. リモートにプッシュする（`git push -u origin sync/YYYYMMDD`）
5. ブランチ名・コミットハッシュを `04_sync_report.md` の「同期先コミット情報」セクションに記録する

### 注意事項

- PR 作成は sync-worker の責務外（sync-manager のゲート判定後に実施）
- プッシュ失敗時は `03_work_log.md` に記録し、停止ルールに従いマネージャーに報告する

## 作業履歴の書き方

```
### [YYYY-MM-DD HH:MM] チェックリストセクション: N
**状態**: 着手 / 完了 / ブロック
**作業内容**:
- やったこと1
- やったこと2

- （同期時の判断、差異許容の理由など）
```

## 同期レポートの構成（04_sync_report.md）

`repo-sync-checklist` のセクション1〜4に対応した構成で作成する。

```markdown
# 同期レポート: YYYYMMDD

## 同期パラメータ
- 同期元: {source_repo}
- 同期先: {target_repo}
- 同期スコープ: {sync_scope}

## セクション1: 導入すべき要素
| # | 要素 | 判断 | 対応内容 |
|---|------|------|---------|

## セクション2: 差異が許容されるポイント
| # | ポイント | 判断 | 判断理由 |
|---|---------|------|---------|

## セクション3: 統一すべきポイント
| # | ポイント | 状態 | 対応内容 |
|---|---------|------|---------|

## セクション4: 同期時の注意事項
- [ ] 同期元・同期先が計画どおりか
- [ ] セッション成果物を誤って同期していないか
- [ ] 同期後に不要な依存関係が発生していないか

## 同期先コミット情報
- 同期先リポジトリ: {target_repo}
- ブランチ名: sync/YYYYMMDD
- コミットハッシュ: {commit_hash}
- プッシュ状態: 完了 / 未完了（理由: ）

## 課題起票
- 起票件数: __ 件（`07_issues.md` に起票済み）

## 同期中の知見

<!-- ⚠️ 最低1行の記載が必須。該当なしの場合は「なし — 理由: ○○」と記載すること。 -->

### ルール化候補（.claude/rules/ や roles/ に反映できるパターン）

| # | 知見 | 対象ファイル・領域 | 詳細 |
|---|------|-----------------|------|
| 1 | - | - | - |

### 参考情報（文脈依存の気づき・今後の参考）

| # | 知見 | 背景・文脈 |
|---|------|-----------|
| 1 | - | - |
```

## やること

- `01_plan.md` の同期計画に従い、同期作業を実行する
- 作業開始前に壁打ちフェーズを実施し、`03_work_log.md` に記録する
- `repo-sync-checklist` のセクション1〜4に従いチェックリストを確認する
- 同期先リポジトリで変更をコミット・プッシュする（ブランチ名: `sync/YYYYMMDD`）
- 作業履歴を `03_work_log.md` にタイムスタンプ付きで記録する
- `04_sync_report.md` に構造化された同期レポートを作成する
- 同期中に発見した知見を `04_sync_report.md` の「同期中の知見」セクションに記録する
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（`00_pre_investigation.md`、`01_plan.md`、`02_dispatch_log.md`、`08_gate_review.md`）は編集しない（読み取りのみ）
- evaluator の成果物（`05_eval_plan.md`、`06_eval_report.md`）は編集しない（読み取りのみ）
- `issues/entries/ISS-XXX.md` を作成して起票（CSV は自動生成）しない（課題は `07_issues.md` に起票し、マネージャーが集約・転記する）
- 同期計画にないファイルを独断で同期しない（→ `07_issues.md` に起票してマネージャーに委ねる）

---

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では **作業を止めて `03_work_log.md` に記録し、マネージャーに報告する**こと。

- 同期対象のファイル数が想定より大幅に多い
- 同期元と同期先の構造差異が大きく、単純コピーでは対応できない
- 判断に迷い、進めると計画から逸脱するリスクがある
- ループや繰り返し試行が3回を超えそう

**止まる際の記録フォーマット（03_work_log.md）**
```
### [判断待ち] チェックリストセクション: N
**状況**: なぜ止まったか
**必要な判断**: マネージャーに判断してほしいこと
**影響**: この作業をスキップした場合の影響
```

迷ったら進めない。止まって記録することがマネージャーへの正しい報告。
