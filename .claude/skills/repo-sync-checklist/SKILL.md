# リポジトリ間同期チェックリスト

> **使い方**: このスキルは `/repo-sync-checklist` で呼び出す。
> リポジトリ間（dev-process-improvement → ai-driven-dev-patterns 等）で構成・ルール・テンプレートを同期する際に、
> 同期作業の L2-worker がこのチェックリストに従って作業を進める。
>
> **作業手順**:
> 1. 同期対象のファイル・ディレクトリを特定する
> 2. 以下の各セクションのチェックリストを順に確認する
> 3. 各項目について「導入」「差異許容」「統一」のいずれかを判断する
> 4. 判断結果を作業ログに記録する

---

## 1. 導入すべき要素

同期先リポジトリに持ち込むべき要素を確認する。

| # | 要素 | dev-process-improvement での実装 | 同期先への適用方針 |
|---|------|-------------------------------|------------------|
| 1 | `sessions/` ディレクトリ | `sessions/initiatives/`, `sessions/triage/` の2階層 | initiatives が存在する場合は同構成を導入。triage は dev-process-improvement 固有のため不要 |
| 2 | `sessions/initiatives/_template/` | 00〜08 の番号付きファイル | initiative テンプレートを持ち込む場合はそのまま適用 |
| 3 | `sessions/initiatives/_archive/` | 完了済み施策の移動先 | 同等のアーカイブディレクトリを作成 |
| 4 | テンプレートコピー手順の改善 | workers/ を空ディレクトリで作成、ワーカーセットは _template/workers/_template/ から直接コピー | triage 構成を持ち込む場合のみ適用 |
| 5 | トリアージ構成 | `triage-standard-policy`（ライフサイクル適用マトリクス）、`triage-manager` + agents（triage-worker, triage-evaluator）、`triage/_template/`（セッションテンプレート） | 同期先に独自のトリアージ実装がある場合は差分を確認し、不足している要素（ポリシー・テンプレート・エージェント定義）を補完導入する |
| 6 | `sessions/` ディレクトリ利用ガイド | CLAUDE.md にセッション種別ごとのディレクトリ用途・使い分けを記載。README にエントリーポイントとして案内 | 同期先の CLAUDE.md または README に `sessions/` 配下の構成と用途（initiatives/ は施策管理、triage/ はトリアージ等）の案内を追加する |

**チェック**:
- [ ] 同期先に `sessions/` ディレクトリ構成を導入したか
- [ ] テンプレートファイルが正しくコピーされたか
- [ ] アーカイブディレクトリが作成されたか
- [ ] トリアージ構成の不足要素を確認・補完したか
- [ ] `sessions/` ディレクトリの利用ガイドが同期先の CLAUDE.md または README に記載されているか

## 2. 差異が許容されるポイント

以下は同期先リポジトリの事情に合わせてカスタマイズしてよい。

| # | ポイント | 理由 |
|---|---------|------|
| 1 | `sessions/triage/` の有無 | トリアージセッションは dev-process-improvement 固有のプロセス改善サイクル。同期先では不要な場合がある |
| 2 | `_archive/` 配下の移動タイミング | アーカイブ移動のタイミングやルールはリポジトリの運用頻度に応じて調整可 |
| 3 | テンプレートファイルの番号付け | プロジェクト規模・ワークフローに応じてファイル構成をカスタマイズしてよい |
| 4 | 課題管理 CSV の列定義 | プロジェクト固有の列を追加・削除してよい |
| 5 | `sessions/triage/_archive/` | triage セッションを持ち込まない場合は不要 |

**チェック**:
- [ ] 差異許容ポイントについて、同期先の事情に合わせた判断を行ったか
- [ ] 判断理由を作業ログに記録したか

## 3. 統一すべきポイント

以下はリポジトリ間で統一する。差異を残さないこと。

| # | ポイント | 理由 | 統一方法 |
|---|---------|------|---------|
| 1 | `sessions/` をトップレベルディレクトリとして使用 | 2リポジトリ間でセッション成果物の配置規則を揃え、相互参照時の混乱を防ぐ | initiative 構造を導入する際は `sessions/initiatives/` を使用する |
| 2 | 施策テンプレートの基本構造（proposal → plan → tasks → report → gate_review） | ライフサイクルの統一により、リポジトリ間で施策の進捗把握方法が共通化される | テンプレートの番号付けとファイル名は統一する |
| 3 | コミットメッセージ規約の `[session-type] category:` 形式 | 2リポジトリ間でコミット履歴の解析方法を統一する | `.claude/rules/commit-message.md` を同期する |
| 4 | `.claude/skills/` のスキル・エージェント構成 | マネージャー=スキル、ワーカー=エージェントの基本パターンを統一する | session-flow-policy の基本構造に従う |
| 5 | `issues/` + CSV の課題管理パターン | 施策横断の課題管理方法を統一する | CSV のコア列（ID, 起票日, カテゴリ, 種別, タイトル, ステータス）は統一する |

**チェック**:
- [ ] 統一すべきポイントが同期先でも同じ構成になっているか
- [ ] コミットメッセージ規約が同期されたか
- [ ] スキル・エージェント構成が統一パターンに従っているか
- [ ] 課題管理の CSV コア列が統一されているか

## 4. 同期時の注意事項

- 同期対象は **構成・ルール・テンプレート** であり、セッション成果物（作業ログ等）は同期しない
- 同期元・同期先は sync-manager の入力解釈で確定される（リポジトリリスト参照）
- 課題管理 CSV は各リポジトリで独立管理のため、CSV 自体は同期対象外

**チェック**:
- [ ] 同期元・同期先が計画どおりか
- [ ] セッション成果物を誤って同期していないか
- [ ] 同期後に不要な依存関係が発生していないか

## 5. 同期結果の検証チェックリスト（7つの検証観点）

同期作業の完了後、sync-evaluator が以下の7つの観点で同期結果を検証する。
各観点はパラメータ化されており、`{source_repo}`, `{target_repo}`, `{sync_scope}` を実際の値に置き換えて使用する。

### 5.1 ファイル存在性の検証

同期対象のファイルが `{target_repo}` に正しく存在するかを確認する。

- **検証内容**: `{sync_scope}` に含まれる全ファイルが `{target_repo}` に存在する
- **検証方法**: `{source_repo}` のファイルリストと `{target_repo}` のファイルリストを突合する
- **パラメータ**: `{source_repo}`, `{target_repo}`, `{sync_scope}`

**チェック**:
- [ ] `{sync_scope}` の全ファイルが `{target_repo}` に存在するか
- [ ] 不要なファイルが `{target_repo}` に混入していないか

### 5.2 内容一致性の検証

統一すべきポイント（セクション3）のファイル内容が `{source_repo}` と `{target_repo}` で一致しているかを確認する。

- **検証内容**: 統一対象のファイル内容が同一である（差異許容ポイントを除く）
- **検証方法**: diff コマンドまたはファイル内容の比較で差異を検出する
- **パラメータ**: `{source_repo}`, `{target_repo}`

**チェック**:
- [ ] 統一すべきファイルの内容が一致しているか
- [ ] 差異許容ポイント（セクション2）以外に差異がないか

### 5.3 構造整合性の検証

ディレクトリ構造が `{source_repo}` の設計意図に沿って `{target_repo}` に反映されているかを確認する。

- **検証内容**: ディレクトリ階層・ファイル命名規則が統一されている
- **検証方法**: ディレクトリツリーの比較、ファイル名パターンの確認
- **パラメータ**: `{source_repo}`, `{target_repo}`, `{sync_scope}`

**チェック**:
- [ ] ディレクトリ階層が設計意図に沿っているか
- [ ] ファイル命名規則（番号付きプレフィックス等）が統一されているか

### 5.4 参照整合性の検証

ファイル間の参照（パス・リンク）が `{target_repo}` で正しく機能するかを確認する。

- **検証内容**: ファイル内のパス参照・リンクが `{target_repo}` のディレクトリ構造で有効である
- **検証方法**: ファイル内のパス参照を抽出し、参照先の存在を確認する
- **パラメータ**: `{target_repo}`

**チェック**:
- [ ] ファイル内のパス参照が `{target_repo}` で有効か
- [ ] 関連ファイル一覧のパスが正しいか

### 5.5 ポリシー準拠性の検証

同期されたファイルが `{target_repo}` のポリシー（CLAUDE.md、リポジトリ境界ポリシー等）に準拠しているかを確認する。

- **検証内容**: 同期結果がリポジトリのポリシーに違反していない
- **検証方法**: CLAUDE.md のポリシー条項と同期結果を突合する
- **パラメータ**: `{source_repo}`, `{target_repo}`

**チェック**:
- [ ] リポジトリ境界ポリシーに違反していないか
- [ ] 同期方向が正しいか（依存関係の方向が正しいか）

### 5.6 差異許容の妥当性検証

差異を許容した判断（セクション2）が妥当であるかを確認する。

- **検証内容**: 差異許容の各判断に合理的な理由が記録されている
- **検証方法**: `04_sync_report.md` のセクション2の判断理由を確認する
- **パラメータ**: `{source_repo}`, `{target_repo}`, `{sync_scope}`

**チェック**:
- [ ] 差異許容の判断に合理的な理由が記載されているか
- [ ] 許容すべきでない差異が見落とされていないか

### 5.7 連動更新の完全性検証

同期に伴う連動更新（コミットメッセージ規約、workflow.md 等）が完了しているかを確認する。

- **検証内容**: 同期対象ファイルの「関連ファイル一覧」に基づく連動更新が漏れなく実施されている
- **検証方法**: 各同期ファイルの「関連ファイル一覧」を確認し、更新先の状態をチェックする
- **パラメータ**: `{target_repo}`

**チェック**:
- [ ] 関連ファイル一覧に基づく連動更新が完了しているか
- [ ] `docs/workflow.md`（人間向け可視化）が最新状態に更新されているか
