# Claude Code ビルトイン Skills 評価リファレンス

## 概要

Claude Code に同梱されるビルトイン（バンドル）skills の、dev-process-improvement プロジェクトにおける適用可能性を評価した結果をまとめたリファレンス。ビルトイン skills の利用判断や、新規カスタム skill 作成時の参考として参照する。

**評価日**: 2026-03-12
**評価対象バージョン**: Claude Code の公式ドキュメント（https://code.claude.com/docs/en/skills）に基づく
**プロジェクト特性**: Markdown 文書・プロセスルール管理が主体。コード実装は含まない

## 評価サマリー

| Skill | 適用可能性 | 推奨利用場面 |
|-------|-----------|------------|
| `/simplify` | 低 | 本プロジェクトでの積極活用場面なし |
| `/batch` | 中 | `.claude/skills/` や `docs/` の大規模一括更新（条件付き） |
| `/debug` | 中 | セッション失敗・サブエージェント問題のアドホック診断 |
| `/loop` | 低 | 本プロジェクトでの積極活用場面なし |
| `/claude-api` | 対象外 | API 実装を含まないため適用場面なし |

## 各 Skill の詳細評価

### /simplify

**機能概要**: 最近変更したファイルのコード再利用・品質・効率性を3つの並列エージェント（コード再利用、コード品質、効率性）でレビューし、問題を検出・修正する。機能実装やバグ修正後のクリーンアップとして使用する。オプションで `focus on memory efficiency` のように特定の観点を指定できる。

**プロジェクトでの適用可能性**: **低**

本プロジェクトは Markdown 文書とプロセスルール定義が主体であり、/simplify が対象とするコード品質レビュー（再利用性・効率性）の適用場面が極めて限定的。Markdown 文書のリファクタリングや冗長性削減は、汎用的なプロンプトや手動レビューの方が文脈に即した判断ができる。

**具体的な利用シーン**: 適用場面なし。3エージェント並列レビューは、コード変更を伴う施策（シェルスクリプト作成等）が万が一発生した場合にのみ検討の余地がある。

**制約事項**: コード変更を前提とした設計のため、Markdown 文書のみの変更では有効に機能しない可能性が高い。

### /batch

**機能概要**: 大規模なコードベース変更を並列で実行する。コードベースを調査し、5〜30 の独立ユニットに分解して計画を提示。承認後、ユニットごとに独立した git worktree でエージェントを生成し、各ユニットの実装・テスト・PR 作成を行う。git リポジトリが必要。

**プロジェクトでの適用可能性**: **中**

理論的には `.claude/skills/` 配下の全 skills のメタデータフォーマット統一や、`docs/` 配下の文書一括更新に使用できる。ただし、本プロジェクトの文書変更は意味的整合性（スキル間の参照関係、用語の統一等）が重要であり、独立ユニットへの分解が困難なケースが多い。

**具体的な利用シーン**:
- 全 SKILL.md ファイルに新しい frontmatter フィールドを一括追加する場合
- テンプレートファイル群のフォーマット統一
- 大量のファイルに対する機械的な置換（用語変更等）

**制約事項**:
- 文書間の意味的依存関係がある変更には不向き（整合性が壊れるリスク）
- git worktree を使用するため、並列実行中は他の作業と干渉する可能性
- 本プロジェクトの変更規模では 5〜30 ユニットに分解するほどの量が発生しにくい

### /debug

**機能概要**: 現在の Claude Code セッションのデバッグログを読み、トラブルシューティングを行う。オプションで問題の説明を指定すると、分析の焦点を絞ることができる。

**プロジェクトでの適用可能性**: **中**

L2 サブエージェント（worker / evaluator）のセッションが期待通りに動作しない場合の問題診断に有用。特に、サブエージェントがツールの権限問題で停止した場合やスキルのロードに失敗した場合に、デバッグログから原因を特定できる。

**具体的な利用シーン**:
- L2 サブエージェントが予期せず停止・エラーを返した場合の原因調査
- スキルが自動トリガーされない場合の description 検証
- セッション中のツール呼び出しエラーの原因特定

**制約事項**:
- 日常的な利用場面は限定的（問題発生時のアドホック利用）
- セッションのデバッグログが対象のため、セッション外の問題（git の設定等）には使えない

### /loop

**機能概要**: 指定した間隔でプロンプトを繰り返し実行する。cron タスクとしてスケジュールされ、セッションが開いている間動作する。デプロイの監視、PR のウォッチ、別のスキルの定期実行に使用する。

**プロジェクトでの適用可能性**: **低**

PR 状態チェックやブランチ状態確認の定期監視に理論上は使えるが、本プロジェクトのセッションは単発タスク完結型（L1 → L2 → ゲート判定）であり、長時間のセッション維持を前提とする /loop の活用場面がほぼない。

**具体的な利用シーン**: 適用場面なし。仮に CI/CD パイプラインを導入した場合にデプロイ監視で使える可能性はあるが、現時点では該当しない。

**制約事項**:
- セッションが開いている間のみ動作するため、バックグラウンドの継続監視には使えない
- 本プロジェクトのワークフロー（短期セッション）と /loop の前提（長期セッション）が合致しない

### /claude-api

**機能概要**: Claude API リファレンス（Python, TypeScript, Java, Go, Ruby, C#, PHP, cURL）と Agent SDK リファレンス（Python, TypeScript）をロードする。ツール使用、ストリーミング、バッチ、構造化出力、一般的な落とし穴をカバー。`anthropic`、`@anthropic-ai/sdk`、`claude_agent_sdk` のインポートがあるコードでは自動的に起動する。

**プロジェクトでの適用可能性**: **対象外**

本プロジェクト（dev-process-improvement）は API 実装を含まず、anthropic SDK を使用していない。Claude Code の利用はあくまで CLI/SDK 上でのプロセス管理であり、API コーディングの場面がない。

**具体的な利用シーン**: 適用場面なし。

**制約事項**: API コーディングを行うリポジトリ（ai-driven-dev-patterns 等）では有用な可能性があるが、本施策のスコープ外。

## 総合方針

### 積極活用する Skills

現時点で積極的に活用を推奨するビルトイン skills はない。本プロジェクトは Markdown 文書・プロセスルール管理が主体であり、ビルトイン skills はコード変更を主な対象として設計されているため、フィット度が低い。

### 状況に応じて活用する Skills

- **/debug**: セッションの問題診断時にアドホックに使用する。L2 サブエージェントのトラブルシューティングで特に有効
- **/batch**: `.claude/skills/` の大規模一括更新（メタデータフォーマット統一等）が発生した場合に検討する。ただし文書間の意味的整合性が必要な変更には使用しない

### 活用しない Skills

- **/simplify**: コード主体のプロジェクト向け。本プロジェクトでは不要
- **/loop**: 長期セッション前提の機能。本プロジェクトの単発タスク完結型ワークフローと合致しない
- **/claude-api**: API 実装を含まないため対象外

### 再評価の推奨

ビルトイン skills は Claude Code のバージョンアップで機能追加・変更される可能性がある。以下のタイミングで再評価を推奨する:
- Claude Code のメジャーバージョンアップ時
- 新しいビルトイン skill が追加された場合
- プロジェクトの性質が変わった場合（コード実装を含む施策が増えた等）
