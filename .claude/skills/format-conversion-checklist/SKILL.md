# フォーマット変換の内容同等性チェックリスト

> **このファイルの位置づけ**
> フォーマット変換（構造変更・形式変更）を行うタスクで、変換前後の内容同等性を検証するためのチェックリスト。
> 「フォーマット変換のみ（内容・ロジック変更なし）」と指示されたタスクで、実用的な参照資料や具体例が意図せず失われることを防止する。
> dev-process-improvement / ai-driven-dev-patterns 両リポジトリに適用する。

---

## 省略可/不可の分類基準

フォーマット変換時に要素を省略してよいかどうかは、以下の判断基準で決定する。

**判断基準**: 「この要素がなくなった場合、読者は同じ作業を再現できるか？」

### 省略不可（必ず保持する要素）

| カテゴリ | 具体例 |
|---------|--------|
| 具体的なコード例・コマンド例 | 設定ファイルのスニペット、CLI コマンド、スクリプト例 |
| 数値・閾値・具体的パラメータ | 行数制限、件数上限、タイムアウト値 |
| 手順の順序・依存関係 | 「A の前に B を実行」等のフロー制約 |
| 条件分岐・例外処理の記述 | 「〜の場合は」「〜を除く」等の条件付きロジック |
| 参照先・リンク | 他ファイルへのパス参照、外部ドキュメントへのリンク |
| テーブル・マトリクスの行と列 | 対応表、マッピング、分類テーブルの全行 |
| 注意事項・禁止事項 | 「〜してはいけない」「〜に注意」等の警告 |

### 省略可（変換時に省略・統合してよい要素）

| カテゴリ | 条件 |
|---------|------|
| 装飾的な表現 | 読みやすさのための修辞であり、情報を含まないもの |
| 冗長な繰り返し | 同一内容が異なる表現で重複している場合（1つを残す） |
| メタ的なコメント | 「このセクションでは〜を説明します」等のナビゲーション文 |
| 旧フォーマット固有の構造要素 | 変換先フォーマットで対応する構造に置き換え済みの場合 |

## 同等性検証手順

フォーマット変換タスクでは、以下の手順で変換前後の同等性を検証する。
割合閾値は使用しない（20% 等の行数差閾値は不適切）。変換時は常に全要素を検証する。

### ステップ 1: 変換前の要素棚卸し

変換前のファイルから、以下のカテゴリに分類して要素を列挙する。

- [ ] **構造要素**: セクション見出し、テーブル、リスト項目の数を記録する
- [ ] **コード・コマンド例**: コードブロック、インラインコードの数と内容を記録する
- [ ] **参照・リンク**: ファイルパス参照、URL、相互参照の数と対象を記録する
- [ ] **条件・ルール**: 条件分岐、制約、禁止事項の数と内容を記録する
- [ ] **データ**: 数値、閾値、具体的パラメータを記録する

### ステップ 2: 変換後の要素照合

変換後のファイルと変換前の棚卸し結果を突合し、以下を確認する。

- [ ] 構造要素が全て変換先に存在する（見出しの統合は可、削除は不可）
- [ ] コード・コマンド例が全て保持されている（フォーマット変更は可、内容変更は不可）
- [ ] 参照・リンクが全て保持されている（パス形式の変更は可、リンク切れは不可）
- [ ] 条件・ルールが全て保持されている（表現の変更は可、条件の変更・削除は不可）
- [ ] データが全て保持されている（表記の変更は可、値の変更は不可）

### ステップ 3: 差分の分類と判断

ステップ 2 で検出された差分を以下に分類する。

| 分類 | 判断 | アクション |
|------|------|-----------|
| フォーマット差異（表現変更のみ） | 許容 | 記録のみ |
| 意図的な省略（省略可の要素） | 許容 | 省略理由を記録 |
| 意図しない欠落（省略不可の要素） | 不許容 | 変換先に復元する |
| 内容の変更（値・条件の変更） | 不許容 | 変換先を元に戻す（内容変更は別タスクで行う） |

## 検証記録テンプレート

フォーマット変換タスクの完了時に、以下のテンプレートを `03_work_log.md` に記録する。

```markdown
### フォーマット変換 同等性検証記録

**変換対象**: <変換元ファイルパス> → <変換先ファイルパス>
**変換種別**: <構造変更 / 形式変更 / 統合 / 分割>

#### 要素棚卸し結果

| カテゴリ | 変換前の数 | 変換後の数 | 差分 |
|---------|-----------|-----------|------|
| 構造要素（見出し・テーブル・リスト） | - | - | - |
| コード・コマンド例 | - | - | - |
| 参照・リンク | - | - | - |
| 条件・ルール | - | - | - |
| データ（数値・閾値） | - | - | - |

#### 差分の分類

| # | 差分内容 | 分類 | 判断 | 備考 |
|---|---------|------|------|------|
| 1 | - | - | - | - |

#### 検証結果
- **判定**: 同等 / 非同等（要修正）
- **修正箇所**: （非同等の場合に記載）
```

## 使い方

### L1 マネージャー（02_tasks.md 作成時）

フォーマット変換を含むタスクでは、変換タスクの完了条件に同等性検証の実施を含める:

```markdown
| T-XXX | <ファイル名>をフォーマット変換する | フォーマット変換チェックリスト（format-conversion-checklist スキル）に従い同等性検証が完了している | 🔴 | ⬜ |
```

### L2 ワーカー（フォーマット変換タスク完了前）

フォーマット変換タスクを実施した後、コミット前に以下を行う:

1. 本チェックリストの「同等性検証手順」のステップ 1〜3 を実施する
2. 「検証記録テンプレート」を使って検証結果を `03_work_log.md` に記録する
3. 「意図しない欠落」や「内容の変更」が検出された場合は、変換先ファイルを修正してから再検証する

### L2 評価者（評価時）

フォーマット変換を含む施策の評価では、以下の観点で検証する:

1. ワーカーが本チェックリストに従って検証記録を残しているか
2. 検証記録の「要素棚卸し結果」が実態と一致しているか（サンプル確認）
3. 「省略不可」に分類される要素が変換先に保持されているか
