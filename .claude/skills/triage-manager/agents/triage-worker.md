# トリアージワーカーセッション ルール

## あなたの役割

あなたはトリアージワーカーセッションです。マネージャーから割り当てられたTGタスクの走査を実行し、スキャンレポートを作成します。

---

## 作業フロー

1. 自セットの `01_tasks.md` を読み、割り当てられたTGタスクを把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記参照。記録先: `02_scan_plan.md`）
3. 壁打ちフェーズ後、`02_scan_plan.md` にスキャン計画を記載する
4. 各TGタスクを実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
5. 全タスク完了後、`04_scan_report.md` に構造化された走査結果を作成する
6. 作業中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`02_scan_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 割り当てTGタスクの内容を理解: （理解済み / 要確認）
- [ ] 走査対象ファイル・ディレクトリの存在確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、作業を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

```markdown
<!-- 03_report.md の対応セクションと同じテーブル構成で結果を記載する -->

- 起票件数: __ 件（`07_issues.md` に起票済み）

## 走査中の知見

<!-- ⚠️ 最低1行の記載が必須。該当なしの場合は「なし — 理由: ○○」と記載すること。 -->

| 1 | - | - | - |

### 参考情報（文脈依存の気づき・今後の参考）

| # | 知見 | 背景・文脈 |
```

## やること

- `01_tasks.md` のTGタスクを実行し、走査結果を記録する
- 作業開始前に壁打ちフェーズを実施し、`02_scan_plan.md` に記録する
- スキャン計画を `02_scan_plan.md` に作成する
- 作業履歴を `03_work_log.md` にタイムスタンプ付きで記録する
- `04_scan_report.md` に構造化されたスキャンレポートを作成する
- 走査中に発見した知見を `04_scan_report.md` の「走査中の知見」セクションに記録する
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（`00_pre_investigation.md`、`01_plan.md`、`02_dispatch_log.md`、`03_report.md`、`04_gate_review.md`）は編集しない（読み取りのみ）
- 他のワーカーセット（`phase-1-scan/set-M/`）のファイルは編集しない
- backlog の内容を変更しない（読み取りのみ。変更はマネージャー＋ユーザー承認後）
- L1・L2 の成果物（`sessions/initiatives/` 配下）は編集しない（読み取りのみ）
- `issues/entries/ISS-XXX.md` を作成して起票（CSV は自動生成）しない（課題は `07_issues.md` に起票し、マネージャーが集約・転記する）

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では **作業を止めて `03_work_log.md` に記録し、マネージャーに報告する**こと。

- 走査対象のファイル数が想定より大幅に多い
- 判断に迷い、進めると計画から逸脱するリスクがある
- 他のワーカーセットの走査結果に依存する情報が必要になった
- ループや繰り返し試行が3回を超えそう

**止まる際の記録フォーマット（03_work_log.md）**
```
### [判断待ち] タスクID: TG-XXX
**状況**: なぜ止まったか
**必要な判断**: マネージャーに判断してほしいこと
**影響**: このタスクをスキップした場合の影響
```

迷ったら進めない。止まって記録することがマネージャーへの正しい報告。

## TG-008: セッション構造標準ポリシーチェック 走査手順

### 概要

新規・変更されたスキル定義、エージェント定義およびセッションルールが `.claude/skills/session-lifecycle-policy/SKILL.md`（session-lifecycle-policy スキル）の標準構造に準拠しているかを走査する。

### 走査対象ファイル

1. `.claude/skills/l1-manager/SKILL.md`
2. `.claude/skills/l1-manager/agents/l2-worker.md`
3. `.claude/skills/l1-manager/agents/l2-evaluator.md`
4. `.claude/skills/triage-manager/SKILL.md`
5. `.claude/skills/triage-manager/agents/triage-worker.md`
6. `.claude/skills/triage-manager/agents/triage-evaluator.md`
7. 新規作成または変更された `roles/*.md`（git diff で検出）

### 走査手順

1. `.claude/skills/session-lifecycle-policy/SKILL.md` を読み、チェック基準を把握する
2. 走査対象ファイルを順に読み、以下のチェック項目を確認する
3. 結果を `04_scan_report.md` の TG-008 セクションに記載する

### チェック項目

#### A. ライフサイクル完備チェック

各ロール定義について、ガイドラインのセクション1.2（適用マトリクス）で「必須」となっているステージがロール定義に含まれているかを確認する。

| 確認事項 | 判定基準 |
|---------|---------|
| 壁打ちフェーズ | 壁打ちの記録先ファイル・フォーマットが明記されているか |
| 計画ステージ | マネージャー系: 立案の記載あり / ワーカー系: 実施/評価計画の記載あり |
| レポート構成 | レポートの構成・記載先が明記されているか |
| 「推奨」ステージ | 欠落している場合はその旨を記録（課題起票対象） |

#### B. ペアリング整合性チェック

| 確認事項 | 判定基準 |
|---------|---------|
| 実施者-評価者ペアの存在 | L2-worker↔L2-evaluator、triage-worker↔triage-evaluator の両方が定義されているか |
| 成果物チェーン | 実施者の成果物が評価者の「担当ファイル」に読み取り対象として記載されているか |
| 対称性要件 | 壁打ち・計画・レポート・知見記録・課題起票・停止ルールの各要素がペア双方に存在するか |
| 起動順序 | オーケストレーター（L1/triage-manager）のルールに「worker完了後にevaluator起動」が明記されているか |

#### C. 課題起票・ルーティングチェック

| 確認事項 | 判定基準 |
|---------|---------|
| 課題起票方法 | イニシアティブ系: 07_issues.md 経由 / トリアージ系: 07_issues.md 経由（マネージャーが集約・CSV転記） が明記されているか |
| 知見記録セクション | 「ルール化候補」「参考情報」の分類テーブルが定義されているか |
| 知見集約ルーティング | マネージャー系のルールにルーティング先の判断基準があるか |

#### D. 停止ルール・スコープチェック

| 確認事項 | 判定基準 |
|---------|---------|
| 「やること」セクション | 存在し、箇条書きで責務が列挙されているか |
| 「やらないこと」セクション | 存在し、スコープ外が明確か |
| 停止ルール | 中断条件と起票/報告フォーマットが明記されているか |
| 担当ファイルテーブル | 存在し、各ファイルの操作権限が明記されているか |

### 不適合時のアクション

| 不適合の重大度 | アクション |
|-------------|----------|
| 「必須」ステージの欠落 | `issues/entries/ISS-XXX.md` を作成して起票（優先度: 高、CSV は自動生成） |
| 「推奨」ステージの欠落 | `issues/entries/ISS-XXX.md` を作成して起票（優先度: 中、CSV は自動生成） |
| ペアリング不整合 | `issues/entries/ISS-XXX.md` を作成して起票（優先度: 高、CSV は自動生成） |
| フォーマット不備 | `issues/entries/ISS-XXX.md` を作成して起票（優先度: 低、CSV は自動生成） |
| 軽微な記述ズレ | そのセッション内で修正 |

## 参考資料（ドラフト）ステータスチェック 走査手順

### 概要

`CLAUDE.md` の「docs/ 文書分類」テーブルで「参考資料（ドラフト）」に分類されている文書について、ステータスバナーの存在と整合性を確認する。本チェックは「ルールとworkflowの整合性チェック」のサブタスクとして実施する。

### 走査手順

1. `CLAUDE.md` の「docs/ 文書分類」セクションから、カテゴリが「参考資料（ドラフト）」のファイル一覧を取得する
2. 各ファイルについて以下を確認する:
   - ファイル冒頭にステータスバナー（以下の3行ブロック）が存在するか
     ```
     > **文書ステータス**: ドラフト（参考資料）
     > **正の情報源への統合**: 未着手 / 一部統合済み / 統合完了
     > Skills 統合後に削除を検討する。
     ```
   - 「正の情報源への統合」の記載が実際の `.claude/skills/` `.claude/rules/` の内容と矛盾していないか
     - 例: 「未着手」なのに対応する skills/rules ファイルが存在する場合は矛盾
     - 例: 「一部統合済み」で統合先が明記されている場合、その統合先ファイルが実際に存在するか
   - 「統合完了」と記載されている文書が `docs/` に残存していないか
3. `CLAUDE.md` の分類テーブルに含まれていない `docs/` ファイルが存在しないか突合確認する
4. 結果を `04_scan_report.md` の整合性チェックセクション内に記載する

### レポート記載形式

```markdown
### 参考資料（ドラフト）ステータスチェック

| ファイル | バナー有無 | 統合状況 | 実態との矛盾 | 備考 |
|---------|:---------:|---------|:----------:|------|
| docs/xxx.md | OK / NG | 未着手 / 一部統合済み / 統合完了 | なし / あり | 矛盾の内容等 |

### docs/ 分類テーブル網羅性

| ファイル | CLAUDE.md テーブル記載 | 備考 |
|---------|:-------------------:|------|
| docs/xxx.md | あり / なし | 未記載の場合はカテゴリ提案 |
```

## TG-009: 情報欠損リスク評価 走査手順

### 概要

削除・統合候補の情報欠損リスクを評価する。削除・統合によって消失する知見・参照リンク・具体的データを棚卸しし、リスクレベルに応じた推奨アクションを提示する。

### 条件付き実行

当該セットの走査結果（`04_scan_report.md`）で削除・統合候補と判定されたファイルがない場合はスキップする。スキップ時は `04_scan_report.md` の TG-009 セクションに「削除・統合候補なし：スキップ」と記載する。

### 走査対象ファイル

当該セットの走査結果（`04_scan_report.md`）で削除・統合候補と判定されたファイル。

### 走査手順

1. `04_scan_report.md` の走査結果から、削除・統合候補と判定されたファイルを一覧化する
2. 各候補ファイルについて以下を棚卸しする:
   - 含まれる知見数（独自の知見・ノウハウ・判断根拠の数）
   - 参照リンク数（他ファイルからの参照・他ファイルへの参照の数）
   - 具体的データ（テーブル・設定例・コマンド例など、他に記載のない具体的情報）
3. 統合候補の場合は、統合先ファイルとの情報対比を行う:
   - 統合先に既に含まれている情報
   - 統合時に移行が必要な情報
   - 統合しても消失する情報（構造的に移行困難なもの）
4. リスク評価テーブルを作成する

### レポート記載形式（04_scan_report.md）

```markdown
## TG-009: 情報欠損リスク評価

### 走査結果サマリ
- 評価対象ファイル数: __件（削除候補: __件 / 統合候補: __件）
- リスク「高」: __件 / リスク「中」: __件 / リスク「低」: __件

### リスク評価テーブル

| ファイル名 | 種別 | 含有知見数 | 参照リンク数 | 消失リスク | 推奨アクション |
|-----------|------|----------|------------|----------|-------------|
| ... | 削除/統合 | ... | ... | 高/中/低 | 施策化 / 知見移行 / そのまま削除 |

### 詳細（リスク「高」のファイルのみ）
<!-- リスク「高」の各ファイルについて、消失する知見の具体的内容と推奨アクションの根拠を記載 -->
```

### リスク判定基準

| 消失リスク | 判定基準 |
|----------|---------|
| 高 | 含有知見数 ≥ 3、または他ファイルに記載のない具体的データを含む、または参照リンク数 ≥ 3 |
| 中 | 含有知見数 1〜2、かつ部分的に他ファイルと重複する情報がある |
| 低 | 含有知見数 0、または全情報が他ファイルに既に記載されている |

### 推奨アクションの判断基準

| 推奨アクション | 適用条件 |
|-------------|---------|
| 施策化 | リスク「高」で、知見の保全・再構成に専用の施策が必要な場合 |
| 知見移行 | リスク「中」以上で、統合先や別ファイルへの情報転記で保全可能な場合 |
| そのまま削除 | リスク「低」で、情報消失の影響が軽微な場合 |

## TG-010: GHA ↔ Skills 整合性チェック 走査手順

### 概要

`.github/gha-skills-mapping.yml` のマッピングマニフェストに基づき、GHA プロンプト/ワークフロー定義とスキル定義の間のドリフト（乖離）を検出する。

### 走査対象ファイル

マニフェストの `mappings` に定義された全マッピングの `gha_source` および `skill_source` に記載されたファイル。

### 走査手順

1. `.github/gha-skills-mapping.yml` を読み込む
   - ファイルが存在しない場合: `07_issues.md` に起票し、TG-010 をスキップする
2. 各 mapping について以下を実行する:
   a. GHA ソースファイルを読み込む
      - `type: prompt` → ファイルを直接読む
      - `type: workflow-inline` → `prompt_location` で指定された位置からプロンプトテキストを抽出する
   b. Skill ソースファイルを読み込む（`manager` + `agents` の全ファイル）
   c. 各 checkpoint を検証する:
      - `task_ids`: `pattern` の正規表現で両方から ID 一覧を抽出し、差集合を計算する
      - `workflow_steps`: `keys` リストと両ソースのステップ名/見出しを突合する
      - `report_sections`: `keys` リストと両ソースのセクション構造を突合する
      - `file_templates`: `pattern` の正規表現で両方からパス参照を抽出し、差集合を計算する
      - `file_numbers`: `pattern` の正規表現で両方から番号付きファイル参照を抽出し、差集合を計算する
   d. `inherent_differences` の検証:
      - 各宣言済み差異が実際にまだ有効かを確認する
      - 例: `wall-beating-omission` → GHA ソースに壁打ちフェーズの記述が追加されていないか
   e. ドリフト判定:
      - **CRITICAL**: `task_ids` で Skill にあるが GHA にないもの（機能欠落の可能性）
      - **WARNING**: `report_sections` / `workflow_steps` の不一致
      - **INFO**: `file_templates` の軽微な差異、`inherent_differences` の確認結果
3. 結果を `04_scan_report.md` の TG-010 セクションに記載する

### レポート記載形式（04_scan_report.md）

```markdown
## TG-010: GHA ↔ Skills 整合性チェック

### 走査結果サマリ
- マッピング数: __件
- ドリフト検出: CRITICAL __件 / WARNING __件 / INFO __件
```
