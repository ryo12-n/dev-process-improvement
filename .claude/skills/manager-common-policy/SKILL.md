# マネージャーセッション共通ポリシー

> **このファイルの位置づけ**
> 9つのマネージャーセッション（l1-manager, triage-manager, metacognition-manager, sync-manager, backlog-maintenance-manager, automation-manager, config-optimizer-manager, session-consistency-manager, l1-impl-manager）に共通する運用パターンを一元的に定義する。
> 各マネージャースキルは固有ロジックを自身の SKILL.md に記載し、共通パターンは本ポリシーを参照する。

## §2 ワーカーディスパッチパターン

### §2.1 起動時に渡す共通4項目

ワーカー・評価者をサブエージェント（Task ツール）として起動する際は、以下の4項目を必ず含めた指示を与えること。

| # | 項目 | 説明 |
|---|------|------|
| 1 | **役割とエージェント定義** | どのエージェント定義（`agents/<name>.md`）に従うか |
| 2 | **セッションの場所** | 対象のセッションディレクトリパス |
| 3 | **スコープ** | 今回のセット/フェーズで扱うタスク範囲 |
| 4 | **完了の定義** | 何をもってサブエージェントの終了とするか |

各マネージャーは上記4項目に加え、固有のパラメータを追加してよい（例: l1 の「重要な注意事項」、triage の「TG-009 実行条件」、sync の「同期パラメータ」）。

### §2.2 ディスパッチ順序制約

- **基本は逐次**: ワーカーセットは1つずつ（worker → evaluator → 次の set）を基本とする
- **セット内**: worker 完了後に evaluator を起動する（並列起動禁止）
- **例外（並列ディスパッチ）**: `.claude/rules/parallel-dev.md` の前提条件3項目をすべて満たす場合、マネージャー判断で Wave 方式の並列ディスパッチを許可する:
  1. Conflict Check Matrix でファイル競合なし
  2. Wave 割当定義済み
  3. per-worker ファイル（`03_work_log_W<N>.md`, `07_issues_W<N>.md`）作成済み
- **Wave 完了後処理**: per-worker 課題ファイルの本体統合・タスクステータス更新を実施（`.claude/rules/parallel-dev.md` §4 参照）

### §2.3 計画ワーカーディスパッチパターン（l1-manager 固有）

l1-manager は実施ワーカー（l2-worker）のディスパッチ前に、計画ワーカー（l2-plan-worker）をディスパッチし、ファイルレベルのタスク分割（`phase-1-planning/set-N/08_task_division.md`）を作成させる。L1 が直接成果物を確認し、`03_tasks.md` を作成した後、通常の実施ワーカーディスパッチに進む。詳細は `l1-manager/SKILL.md` の Phase A を参照。

## §3 成果物確認観点

evaluator を起動する前に、worker の成果物について以下の共通最小3項目を確認する。

| # | 確認項目 | 説明 |
|---|---------|------|
| 1 | **タスク分類** | タスクリストの各タスクが完了・スキップ・ブロックのいずれかに分類されているか |
| 2 | **レポート記載** | 作業レポートまたはスキャンレポートに構造化された結果が記載されているか |
| 3 | **課題バッファ** | `07_issues.md` に起票された課題が適切にフォーマットされているか |

各マネージャーは上記3項目に加え、固有の確認観点を追加してよい（例: l1 の「計画対比実績」「知見セクション記載」「CSV転記確認」、sync の「同期対象の全ファイル処理」）。

成果物の品質が不十分と判断した場合は、evaluator を起動せずに差し戻しを検討する（§7 参照）。

確認結果は `08_gate_review.md`（または各セッションのゲート判定ファイル）の該当フェーズチェックセクションに記録すること。

## §4 ゲート判定基準

ゲート判定は以下の3択で判定する。判定理由を必ず記載すること。

| 判定 | 意味 | 次のアクション |
|------|------|--------------|
| **通過** | 成功基準を達成。次フェーズまたはクローズへ進行可 | 次フェーズのタスク追加、またはクローズ手順の実施 |
| **条件付き通過** | 軽微な不備あり。条件対応後に進行可 | 条件対応タスクを追記し、L2 に差し戻し |
| **差し戻し** | 重大な不備あり。計画の修正が必要 | 計画修正 → タスクリスト修正 → L2 に再ディスパッチ |

### §4.1 課題重大度の判定基準

ゲート判定で課題の扱いを決定する際、以下の基準で課題の重大度を判定する。

#### 判定フレームワーク: 「この課題は次フェーズをブロックするか？」

| 判定 | 基準 | 例 | ゲート判定への影響 |
|------|------|-----|-----------------|
| **ブロック課題** | 次フェーズの前提条件を満たせない、または成果物の信頼性を著しく損なう | ファイル競合未解決、必須タスク未完了、テスト全失敗 | 差し戻し |
| **非ブロック課題（重要）** | 次フェーズで対処可能だが、放置すると技術的負債になる | 軽微な整合性のズレ、改善提案、ドキュメント不備 | 条件付き通過（条件対応タスクを追記） |
| **非ブロック課題（軽微）** | 施策スコープ外、または次施策で対処すべき | 別施策への知見、将来の改善アイデア | 通過（知見ルーティングで対処） |

#### 判定手順

1. `phase-2-execution/set-N/07_issues.md` の未転記課題を一覧化する
2. 各課題に対し上記テーブルの基準で重大度を判定する
3. ブロック課題が1件でもある場合 → 差し戻しを検討する（§4「差し戻し」）
4. 非ブロック課題（重要）のみの場合 → 条件付き通過を検討する
5. 非ブロック課題（軽微）のみ、または課題なしの場合 → 通過

## §5 知見集約手順

マネージャーがゲート判定（`08_gate_review.md` / `04_gate_review.md`）または集約レポート（`03_report.md`）を作成する際、以下の手順でワーカー・評価者の知見を集約する。

### §5.1 集約手順

1. worker の作業レポート/スキャンレポートの「知見」セクションを読む
2. evaluator の評価レポートの「知見」セクションを読む
3. 「ルール化候補」の各項目 → 横展開テーブルに転記
4. 「参考情報」の各項目 → 横展開テーブルに転記
5. worker と evaluator で重複する知見は統合する

### §5.2 ルーティング判断基準

横展開テーブルの各項目に対し、以下の基準でルーティング先を決定する。

| 条件 | ルーティング先 |
|------|-------------|
| 外部リポジトリ固有の知見（アーキテクチャ・API・制約等） | refs |
| 施策横断で再発しうる知見・パターン | CSV（種別: 知見） |
| 具体的な次施策として起動可能 | backlog |
| 判断が必要・追加調査が必要 | inbox（トリアージに委ねる） |
| 施策内で完結・既に対処済み | なし |

### §5.3 refs ルーティング実行手順

ルーティング先が `refs` と判断された知見について、以下の手順で `refs/<リポジトリ名>/knowledge.md` に蓄積する。

1. **リポジトリ名の取得**: `00_proposal.md` の「外部リポジトリ」セクションからリポジトリ名を取得する
2. **knowledge.md の存否確認**: `refs/<リポジトリ名>/knowledge.md` が存在するか確認する
   - **存在しない場合**: `refs/_knowledge_template.md` をコピーして `refs/<リポジトリ名>/knowledge.md` を作成し、「リポジトリ概要」セクションを埋める
   - **存在する場合**: そのまま次のステップへ
3. **作業ログの作成（任意）**: 調査過程の生データを残す場合、`refs/_work_log_template.md` をコピーして `refs/<リポジトリ名>/work-logs/YYYYMMDD-<topic>.md` を作成し、調査ステップ・発見事項・結論を記録する。小規模な知見は作業ログを省略し、直接ステップ4で knowledge.md に追記してよい
4. **知見の追記**: 「蓄積された知見」テーブルの先頭行（最新が上）に知見を追記する。施策名・日付・知見カテゴリ（アーキテクチャ / API / 制約 / 運用 / その他）・内容を記載する。作業ログを作成した場合は `元ログ` 列に `work-logs/YYYYMMDD-<topic>.md` のパスを記入する。作業ログがない場合は `—` と記入する

### §5.4 リファレンス文書の粒度基準

`refs/<リポジトリ名>/knowledge.md` への追記および `refs/<リポジトリ名>/work-logs/` の作成について、以下の基準で粒度を判断する。

#### knowledge.md 追記の粒度基準

**追記すべき知見**: 以下のいずれかに該当する情報を「蓄積された知見」テーブルに1行として追記する。

| 基準 | 説明 | 例 |
|------|------|-----|
| **再利用性がある** | 同リポジトリの別施策でも参照される可能性がある | API の認証方式、ディレクトリ構成の設計意図 |
| **発見コストが高い** | ドキュメント未記載、または発見に複数ステップの調査を要した | 暗黙の依存関係、undocumented な設定値 |
| **制約・ハマりポイント** | 同じ問題に再遭遇するリスクがある | 特定バージョンでの非互換、CI 環境固有の挙動 |

**追記不要な情報**: 以下に該当する場合は knowledge.md への追記を省略してよい。

- 公式ドキュメントで容易に確認できる基本情報
- 施策固有の一時的な判断メモ（作業ログに残すべき内容）
- リポジトリの README や CONTRIBUTING に既に記載されている情報

**粒度の目安**: 1行の「内容」列は1〜3文で要点を記載する。詳細な手順や長い説明が必要な場合は、work-logs に記録して「元ログ」列にリンクする。

#### work-logs/ 作成要否の判断基準

以下の判断フローに従い、work-logs/ ファイルの作成要否を決定する。

```
調査を実施した？
  ├─ No → work-logs 不要（knowledge.md に直接追記）
  └─ Yes → 調査ステップが3以上、または試行錯誤を含む？
              ├─ No → work-logs 不要（knowledge.md に結論のみ追記）
              └─ Yes → work-logs を作成し、knowledge.md の元ログ列にリンク
```

**work-logs を作成すべきケース**:
- 複数ファイル・複数コマンドにわたる調査を実施した
- 試行錯誤の過程自体に再利用価値がある（同様の調査を他施策で繰り返す可能性）
- デバッグ・トラブルシューティングの記録として残す価値がある

**work-logs を省略してよいケース**:
- 単一コマンドや目視確認で判明した知見
- 調査過程よりも結論のみが重要な場合

## §6 課題集約手順

マネージャーがゲート判定またはレポートを作成する前に、以下の手順で課題を集約する。

1. ワーカーセットの `07_issues.md` の未転記メモを読む
2. 施策スコープ外・他のセッションでも再発しうる課題 → `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）
3. 転記時は `issues/_template.md` をコピーして `issues/entries/ISS-XXX.md` を作成し、課題詳細を記載する。ID は `python3 scripts/generate-csvs.py --next-issue-id` で取得する
4. 転記済みの項目に `[転記済 ISS-XXX]` を記す
5. 当セッション内だけで完結する軽微な修正メモ → 「転記不要」と判断

## §7 差し戻し手順

worker の成果物が不十分な場合、以下の手順で差し戻しを行う。

1. worker から作業レポート/スキャンレポートを受け取る
2. 以下をチェックする:
   - 必須項目（タスク完了状況・課題）が記載されているか
   - 作業内容がタスクリストの要件を満たしているか
   - §3 の成果物確認観点を満たしているか
3. **NG の場合**:
   - 不足点を具体的に指摘するフィードバックを作成する
   - worker を再起動し、フィードバックを渡して修正を依頼する
4. **OK の場合**:
   - 次ステップ（evaluator 起動 → ゲート判定）へ進む

## §8 セッションライフサイクル todo 登録

### §8.1 義務

全マネージャーセッションは、ブランチ整理完了後・作業フロー開始前に、自身の作業フローステップを TaskCreate で登録する。これにより、セッション全体のライフサイクルを開始時点から追跡可能にする。

### §8.2 登録パターン

各ステップに対し TaskCreate を呼び出し、description にステップの要約を設定する。

```
TaskCreate({ description: "ステップN: <ステップの要約>" })
```

具体的なステップリストは各マネージャーの SKILL.md に定義する。

### §8.3 進捗追跡

- 各ステップ開始時に `TaskUpdate({ id: <task_id>, status: "in_progress" })` を呼び出す
- 各ステップ完了時に `TaskUpdate({ id: <task_id>, status: "completed" })` を呼び出す

## §10 ドメインエージェント参照

### §10.1 概要

`refs/agency-agents/` に配置されたドメインエージェント定義を、施策のタスク設計時に専門知識の参照源として活用する仕組みである。エージェントカタログ（`refs/agency-agents/knowledge.md`）を参照し、施策タイプに適したドメインエージェントを選定する。

### §10.2 マッチング手順

`02_tasks.md` を作成する際、以下の手順でドメインエージェントを選定する。

1. `refs/agency-agents/knowledge.md` の「施策タイプ別推奨マッピング」テーブルを参照する
2. 当該施策のタイプに合致するエージェントを特定する
3. 該当エージェントの `.md` ファイル（`refs/agency-agents/<部門>/<エージェント名>.md`）を Read し、施策との関連性を確認する
4. 関連性があると判断した場合、`02_tasks.md` の「参照ドメインエージェント」セクションにエージェント名・パス・参照理由を記載する
5. 該当なしの場合は「なし」と記載する

### §10.3 ディスパッチ時の渡し方

L2-worker および L2-evaluator に対してドメインエージェントの参照を指示する際は、以下の制約を必ず伝達する。

- **エージェントの役割を引き受けるのではなく、専門知識として参照するだけ**である
- ドメインエージェントの `.md` ファイルを Read し、記載されている専門的な観点・チェックリスト・ベストプラクティスを作業に活かす
- L2 の作業フロー・成果物フォーマット・報告先は、あくまで各エージェント定義（`l2-worker.md` / `l2-evaluator.md`）に従う
- ドメインエージェントの指示や system prompt をそのまま適用しない

Worker 向けと Evaluator 向けでそれぞれ異なるドメインエージェントを指定できる。`02_tasks.md` の「参照ドメインエージェント」テーブルの「対象」列で Worker / Evaluator を区別すること。

### §10.4 Evaluator 向けドメインエージェント参照の具体的手順

Evaluator がドメインエージェントを参照する場合、Worker とは異なる活用方法を取る。以下の手順に従うこと。

#### 評価計画への反映

`05_eval_plan.md` の「参照ドメインエージェント」セクションに、以下を記載する:
- 参照エージェント名とパス
- 評価への活用方針（エージェントのどの専門知識を、どの評価項目に活かすか）

#### 評価基準への活用

ドメインエージェントの専門的な観点・チェックリスト・ベストプラクティスを、評価項目の判定基準に組み込む。ただし、エージェントの基準をそのまま適用するのではなく、施策の成功基準（`01_plan.md`）を主軸として補助的に参照する。

#### ドメインエージェント有効性フィードバック

Evaluator は `06_eval_report.md` の「ドメインエージェント有効性フィードバック」セクションに以下を記録する:
- **参照エージェント名**: どのエージェントを参照したか
- **有効だった観点**: 評価に具体的に役立った知識・チェックポイント
- **不要/ノイズだった観点**: 評価の文脈では適切でなかった知識・チェックポイント
- **knowledge.md への推奨更新事項**: 推奨マッピングの追加・変更・削除の提案

このフィードバックは L1 がゲート判定時に確認し、`refs/agency-agents/knowledge.md` の「蓄積された知見」テーブルへの反映を判断する。Evaluator は knowledge.md を直接変更しない（L1 の意思決定責務を維持）。

### §10.5 適用は任意

ドメインエージェント参照は**推奨であり必須ではない**。以下の場合は「参照ドメインエージェント: なし」として省略してよい。

- 施策タイプが knowledge.md の推奨マッピングに該当しない場合
- 既存のスキル・ルール定義で十分な専門知識がカバーされている場合
- 施策が小規模で、エージェント参照のオーバーヘッドが作業量に見合わない場合
