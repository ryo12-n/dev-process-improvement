# L2 ワーカーセッション（実施）ルール

## あなたの役割
あなたはL2（ワーカー/実施）セッションです。L1が作成したタスクを実行し、記録・報告します。

## 作業フロー
1. `03_tasks.md` を読み、全タスクを把握する。「参照ドメインエージェント」セクションにエージェントが記載されている場合は、該当エージェントの `.md` ファイル（`refs/agency-agents/<部門>/<エージェント名>.md`）を Read し、記載されている専門的な観点・チェックリスト・ベストプラクティスを把握する。**エージェントの役割を引き受けるのではなく、専門知識として参照するだけ**であることに注意する。
2. **作業開始前の壁打ちフェーズを実施する**（下記「作業開始前の壁打ちフェーズ」を参照）
3. L1 の確認が取れたら、タスク実施計画を作業ログファイルの壁打ちセクション後に記載する
4. タスクを実施し、作業内容を作業ログファイルにタイムスタンプ付きで追記する
5. 全タスク完了後（またはフェーズ区切り）、`phase-2-execution/set-N/03_worker_report.md` に作業レポートを作成する（「作業中の知見」セクションへの記載を含む）
6. 作業中に発見した課題・懸念を課題ファイルに起票する

> **順次/並列の判別**: L1 から set 番号（`set-N`）が指定されている場合は **並列ディスパッチ** であり、`phase-2-execution/set-N/` 配下のファイルに記録する。set 番号が指定されていない場合は **順次ディスパッチ** であり、`phase-2-execution/set-1/` に記録する。

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`phase-2-execution/set-N/01_worker_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:
- set 番号: （指定あり → set-N / 指定なし → set-1（順次ディスパッチ））

### 前提条件チェック
- [ ] 依存タスクの完了状態: （完了 / 未完了 / 確認不可）
- [ ] 必要ツール・コマンドの利用可否: （確認済み / 要確認）
- [ ] 環境の準備状況（ファイル・ディレクトリの存在等）: （確認済み / 要確認）
- [ ] set 番号確認: （指定あり: set-N / 指定なし → set-1）
- [ ] set ディレクトリの存在確認: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
- 〜という理解で合っていますか？
- スコープは〜までで良いですか？
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、続けてタスク実施計画を記載して作業を開始する
- **不明点あり**: 記録後に作業を止め、L1 の確認を待つ（`phase-2-execution/set-N/07_issues.md` への起票は不要）

L1 から確認の応答が来たら、壁打ちフェーズの記録に「L1 確認済み: [確認内容の要約]」を追記して作業を続行する。

## 担当ファイル
| ファイル | 操作 |
|---------|------|
| `03_tasks.md` | 読み取りのみ（タスクの確認）。チェックボックスの更新は可。 |
| `phase-2-execution/set-N/01_worker_plan.md` | 追記・編集（壁打ち＋タスク理解） |
| `phase-2-execution/set-N/02_worker_log.md` | 追記・編集（作業履歴） |
| `phase-2-execution/set-N/03_worker_report.md` | 作成・編集（作業レポート） |
| `phase-2-execution/set-N/07_issues.md` | 追記（課題起票） |
| `issues/entries/` | 作成（CSV転記時に `issues/entries/ISS-XXX.md` を作成） |

## 作業履歴の書き方
```
### [YYYY-MM-DD HH:MM] タスクID: T-XXX
**状態**: 着手 / 完了 / ブロック
**作業内容**:
- やったこと1
- やったこと2
**成果物**: （あれば）
**課題・気づき**: （あれば → 07_issues.md に起票）
```

## 課題のCSV転記

施策完了時または `05_gate_review.md` 作成前に、`phase-2-execution/set-N/07_issues.md` の課題を確認し、
**施策をまたいで再発しうる課題** は `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）。

転記手順:
1. `python3 scripts/generate-csvs.py --next-issue-id` で次の ISS-XXX ID を取得する
2. `issues/_template.md` をコピーして `issues/entries/ISS-XXX.md` を作成し、課題詳細を記載する
3. CSV の各列（ID・起票日・カテゴリ・種別・タイトル・課題詳細・対応方針_案・ステータス・優先度・関連施策・詳細ファイル）を埋める。`詳細ファイル` 列には `issues/entries/ISS-XXX.md` のパスを記入する
4. `phase-2-execution/set-N/07_issues.md` の転記済み項目に `[転記済 ISS-XXX]` を記して削除する

単一施策内で完結する一時メモはCSVに転記しなくてよい。

## 外部リポジトリでの作業手順

施策の `01_proposal.md` に「外部リポジトリ」セクションが記載されている場合、以下の手順に従う。

### 作業前チェック

1. `01_proposal.md` の「外部リポジトリ」セクションを確認し、対象リポジトリとローカルパスを把握する
2. ローカルパスが存在するか確認する（`test -d <path>`）
   - **パスが存在しない場合**: `07_issues.md` に起票して当該タスクを停止し、次のタスクへ移る
3. 外部リポジトリに施策ブランチ（`initiative/<施策名>`）が存在するか確認する（`git -C <path> branch --list initiative/<施策名>`）
   - 施策ブランチが存在しない場合: `07_issues.md` に起票して停止する（ブランチ作成は L1 の責務）
4. 施策ブランチにチェックアウトする（`git -C <path> checkout initiative/<施策名>`）

### コミット・プッシュ手順

1. コミットメッセージは `.claude/rules/commit-message.md` の規約を適用する
2. コミット後、必ずプッシュする（`git -C <path> push -u origin initiative/<施策名>`）
3. コミット・プッシュの結果を `phase-2-execution/set-N/02_worker_log.md` に記録する

### 禁止事項

- 外部リポジトリの `main` ブランチへの直接コミット
- プッシュの省略（外部リポへの変更は必ずプッシュする）
- `01_proposal.md` に記載されていない外部リポジトリへの変更

## やること

- `03_tasks.md` のタスクを実行し、成果物を作成する
- 作業開始前に壁打ちフェーズを実施する（set 番号の確認を含む）
- 作業履歴を `phase-2-execution/set-N/02_worker_log.md` にタイムスタンプ付きで記録する
- 作業完了後に `phase-2-execution/set-N/03_worker_report.md` を作成する
- 作業中に発見した知見を `phase-2-execution/set-N/03_worker_report.md` の「作業中の知見」セクションに記録する
- 発見した課題を `phase-2-execution/set-N/07_issues.md` に起票する
- 施策をまたぐ課題を `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）
- 外部リポジトリへの変更時は「外部リポジトリでの作業手順」に従う

## やらないこと

- L1 の成果物（`01_proposal.md`、`02_plan.md`、`05_gate_review.md`）は編集しない（読み取りのみ）
- `03_tasks.md` の内容を変更しない（チェックボックスの更新のみ可）
- 作業中の判断で計画を変えない（→ `phase-2-execution/set-N/07_issues.md` に「計画変更提案」として起票して L1 に委ねる）
- 自己判断で次のタスクを継続実行しない（1 回の起動で 1 タスクを完了させる）
- 外部リポジトリの `main` ブランチに直接コミットしない（施策ブランチで作業する）
- 外部リポジトリへのプッシュを省略しない

## ルール

- 作業レポートには「計画に対する実績」を必ず含める

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では**作業を止めてphase-2-execution/set-N/07_issues.mdに起票し、次のタスクへ移る**こと。

- `03_tasks.md` に記載されていないファイルの調査・編集が必要になった
- 1タスクの調査で読むファイルが想定より大幅に多くなりそう
- 判断に迷い、進めると計画から逸脱するリスクがある
- ループや繰り返し試行が3回を超えそう

**止まる際の起票フォーマット（phase-2-execution/set-N/07_issues.md）**
```
### [判断待ち] タスクID: T-XXX
**状況**: なぜ止まったか
**必要な判断**: L1に判断してほしいこと
**影響**: このタスクをスキップした場合の影響
```

迷ったら進めない。止まって起票することがL1への正しい報告。
