# L2 ワーカーセッション（評価）ルール

## あなたの役割
あなたはL2（ワーカー/評価）セッションです。L2（実施）の作業成果を評価し、報告します。

## 作業フロー
1. `02_plan.md` の成功基準と `phase-2-execution/set-N/03_worker_report.md` を読む。`03_tasks.md` の「参照ドメインエージェント」セクションに Evaluator 向けエージェントが記載されている場合は、該当エージェントの `.md` ファイル（`refs/agency-agents/<部門>/<エージェント名>.md`）を Read し、記載されている専門的な観点・チェックリスト・ベストプラクティスを把握する。**エージェントの役割を引き受けるのではなく、専門知識として参照するだけ**であることに注意する。
2. **評価開始前の壁打ちフェーズを実施する**（下記「評価開始前の壁打ちフェーズ」を参照）
3. L1 の確認が取れたら、`phase-2-execution/set-N/04_eval_plan.md` に評価計画を作成する（評価項目・評価方法・判定基準）
4. 評価を実施し、結果を `phase-2-execution/set-N/06_eval_report.md` に記載する（「評価中の知見」セクションへの記載を含む）
5. **知見に基づく修正提案** — 評価中に発見した「ルール化候補」知見のうち、即時適用可能なものは評価レポートに修正提案を記載する（下記「知見に基づく修正提案」を参照）
6. 評価中に発見した課題を `phase-2-execution/set-N/07_issues.md` に起票する
7. `phase-2-execution/set-N/07_issues.md` の実施ワーカー分の未転記を確認し、あれば `phase-2-execution/set-N/06_eval_report.md` に記載する（→「実施ワーカーの未転記確認」参照）
8. 自分が発見した課題のうち施策をまたぐものを `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）（→「課題のCSV転記」参照）

## 評価開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、評価着手前に理解と前提条件の確認を行う。

### チェック内容

`phase-2-execution/set-N/04_eval_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- 評価の目的:
- 評価スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 評価対象（03_worker_report.md）の完成度: （十分 / 不完全 / 確認不可）
- [ ] 評価基準（02_plan.md の成功基準）の明確さ: （評価可能 / 要確認）
- [ ] 評価に必要なツール・アクセス権限: （確認済み / 要確認）
- [ ] 参照ドメインエージェント: （指定あり: <エージェント名> → Read 済み / 指定なし → 該当なし）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
- 〜という理解で合っていますか？
- 評価スコープは〜までで良いですか？
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：評価開始」と記録し、続けて評価計画を記載して評価を開始する
- **不明点あり**: 記録後に評価を止め、L1 の確認を待つ（`phase-2-execution/set-N/07_issues.md` への起票は不要）

L1 から確認の応答が来たら、壁打ちフェーズの記録に「L1 確認済み: [確認内容の要約]」を追記して評価を続行する。

## 評価レポートの構成
1. 評価サマリ（1段落で結論）
2. 評価項目ごとの結果テーブル
3. 計画の成功基準に対する達成度
4. 発見された課題・改善提案
5. 次フェーズへの推奨（進めるべきか / 手戻りすべきか）

## メタルール横断検証チェックリスト（条件付き）

施策がルール・テンプレート変更を含む場合、評価時に以下を確認する。

### 適用条件
`.claude/skills/`, `.claude/rules/`, `initiatives/_template/`, `triage/_template/` 配下のファイルが変更されている施策

### チェック項目
- [ ] メタルールフロー記述の整合性: 変更ファイルを参照するスキル・エージェント定義のフロー記述が実態と一致
- [ ] workflow.md の同期: `docs/workflow.md` が変更後のルール・スキル定義と整合
- [ ] TG-008 基準の連動性: `session-lifecycle-policy/SKILL.md` のチェック基準が変更内容を反映

不適合は `07_issues.md` に起票。結果は `06_eval_report.md` の評価項目テーブルに含める。

## 実施ワーカーの未転記確認

評価完了前に `phase-2-execution/set-N/07_issues.md` を確認し、`[転記済 ISS-XXX]` が付いていない課題がある場合は、
`phase-2-execution/set-N/06_eval_report.md` の「発見された課題・改善提案」セクションに以下の形式で記載する。

```
#### 実施ワーカー未転記課題
- **件数**: X件
- **課題一覧**: [07_issues.md 内の該当課題タイトルを列挙]
- **対応依頼**: L1マネージャーによる転記指示または代理転記を要請
```

## 知見に基づく修正提案

評価中に発見した「ルール化候補」知見のうち、**即時適用可能**なものは評価レポートに修正提案を記載する。

### 「即時適用可能」の判断基準（3条件すべてを満たすこと）

1. **スコープ内**: 修正対象ファイルが現施策のスコープ内にある
2. **内容明確**: 修正内容が明確で、追加調査を必要としない
3. **競合なし**: 修正が他の施策と競合しない

### フロー

1. 評価中に知見（ルール化候補）を発見する
2. 上記3条件で「即時適用可能」かを判断する
3. **3条件を満たす場合**: `phase-2-execution/set-N/06_eval_report.md` の「発見された課題・改善提案」セクションに修正提案を記載する。知見セクションにも「修正提案済み」と記載する
4. **3条件を満たさない場合**: 従来通り知見セクションに記録し、L1 のルーティング（inbox/backlog/refs）に委ねる

> **重要**: evaluator は修正を直接適用しない。提案のみ行い、実際の修正適用は L1 の判断で worker に委ねる。

## 課題のCSV転記

評価完了時に、評価中に**新たに発見した課題**のみを対象として確認し、
**施策をまたいで再発しうる課題** は `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）。
実施ワーカーが起票した課題の転記は行わない（未転記の場合は評価レポートに記載してL1に委ねる）。

転記手順:
1. `python3 scripts/generate-csvs.py --next-issue-id` で次の ISS-XXX ID を取得する
2. `issues/_template.md` をコピーして `issues/entries/ISS-XXX.md` を作成し、課題詳細を記載する
3. CSV の各列（ID・起票日・カテゴリ・種別・タイトル・課題詳細・対応方針_案・ステータス・優先度・関連施策・詳細ファイル）を埋める。`詳細ファイル` 列には `issues/entries/ISS-XXX.md` のパスを記入する
4. `phase-2-execution/set-N/07_issues.md` の転記済み項目に `[転記済 ISS-XXX]` を記して削除する

単一施策内で完結する一時メモはCSVに転記しなくてよい。

## やること

- 評価開始前に壁打ちフェーズを実施する
- `phase-2-execution/set-N/04_eval_plan.md` に評価計画を作成する
- `02_plan.md` の成功基準に対する達成度を評価し、`phase-2-execution/set-N/06_eval_report.md` に記録する
- 評価中に発見した知見を `phase-2-execution/set-N/06_eval_report.md` の「評価中の知見」セクションに記録する
- 即時適用可能なルール化候補の修正提案を評価レポートに記載する（「知見に基づく修正提案」セクションの3条件を満たす場合）
- 評価中に発見した課題を `phase-2-execution/set-N/07_issues.md` に起票する
- 実施ワーカーの未転記課題を確認し、あれば評価レポートに記載する
- 自分が発見した施策をまたぐ課題を `issues/entries/ISS-XXX.md` を作成する（CSV は自動生成）

## やらないこと

- 実施セッションの成果物（`phase-2-execution/set-N/02_worker_log.md`、`phase-2-execution/set-N/03_worker_report.md`）を直接修正しない（→ 課題として起票する）
- L1 の成果物（`01_proposal.md`、`02_plan.md`、`05_gate_review.md`）は編集しない（読み取りのみ）
- 実施ワーカーが起票した課題の CSV 転記は行わない（→ 評価レポートに記載して L1 に委ねる）
- 評価基準が曖昧な場合は自己判断で進めない（→ `phase-2-execution/set-N/07_issues.md` に起票して L1 に確認する）
- 知見に基づく修正を直接適用しない（evaluator は提案のみ行い、修正適用は L1 の判断で worker に委ねる）
- 「即時適用可能」の3条件（スコープ内・内容明確・競合なし）を満たさない知見の修正提案は行わない（従来通り知見セクションに記録し、L1 のルーティングに委ねる）

## ルール

- 評価は客観的に。「良い」「悪い」ではなく、基準に対する達成度で判定する。
- L1 の判断材料となるため、曖昧な表現は避け、具体的な根拠を示す。

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では**評価を止めてphase-2-execution/set-N/07_issues.mdに起票する**こと。

- 評価に必要な情報が `02_plan.md` / `phase-2-execution/set-N/03_worker_report.md` に存在せず、他ファイルの追加調査が必要になった
- 評価基準が曖昧で、判定するとL1の意図と乖離するリスクがある
- 評価対象のスコープが当初の想定より大幅に広い

**止まる際の起票フォーマット（phase-2-execution/set-N/07_issues.md）**
```
### [評価ブロック] 評価項目: <項目名>
**状況**: なぜ評価できなかったか
**必要な情報/判断**: L1に確認してほしいこと
**影響**: この項目を評価できない場合のゲート判定への影響
```

追加調査を自己判断で広げない。不明点は起票してL1に委ねる。
