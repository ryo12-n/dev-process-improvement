## backlog と課題管理 CSV の関係

本リポジトリでは **backlog** と **課題管理 CSV** という2つの管理システムを運用しているが、これらは異なる目的で使い分けられており、重複ではない。

### backlog（`backlog/*.md`）

- **目的**: トリアージで承認された施策候補の管理
- **単位**: 1施策 = 1ファイル（`.md`）
- **内容**: 施策名・優先度・ステータス・課題背景・期待効果・補足情報
- **ライフサイクル**: inbox からトリアージで承認 → backlog に追加 → initiative 開始時に `git rm` で削除（内容は `00_proposal.md` に引き継がれる）
- **管理者**: トリアージセッション（追加）、L1マネージャー（施策開始時の削除）

### 課題管理 CSV（`プロセス改善_課題管理.csv`）

- **目的**: 施策横断の課題トラッキング
- **単位**: 1課題 = 1行（`ISS-XXX` 番号で管理）
- **内容**: ID・起票日・カテゴリ・種別・タイトル・課題詳細・対応方針案・ステータス・優先度・関連施策
- **ライフサイクル**: 施策実施中に発見 → `07_issues.md` に一時記録 → 施策横断の課題は CSV に転記 → 対応完了でクローズ
- **管理者**: L2ワーカー（起票・転記）、L1マネージャー（ステータス更新）、トリアージセッション（棚卸し）

### 使い分けの基準

| 観点 | backlog | 課題管理 CSV |
|------|---------|-------------|
| 粒度 | 施策（やるべきこと） | 課題（気づき・問題・改善提案） |
| スコープ | 単一施策 | 施策横断 |
| 採番 | ファイル名 | ISS-XXX |
| 状態管理 | ステータス欄（候補 / initiative 開始済） | ステータス列（起票 / 対応済 / クローズ 等） |

## 運用ルール

1. **新規 backlog 追加時**: backlog ファイル（`.md`）を作成したら、本 README.md のテーブルにも行を追加すること
2. **initiative 開始時**: 施策を `initiatives/` で開始したら、テーブル内の該当行のステータスを「initiative 開始済」に更新すること
3. **backlog ファイル削除時**: `git rm` で backlog ファイルを削除した場合は、テーブルから該当行を削除すること
