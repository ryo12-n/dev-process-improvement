## 課題・背景

現状の同期セッション（sync-manager）では、事前調査フェーズで毎回「両リポジトリの .claude/ を広く探索 → 同期候補を特定 → 同期方針を決定」している。この方式には以下の問題がある:

1. **調査コストが高い**: 毎回ゼロからの探索が必要で、Explore エージェントの起動に時間がかかる
2. **差分の起点が不明確**: 前回同期以降の変更を特定するのに git log の目視確認に頼っている
3. **見落としリスク**: 同期対象の網羅性が調査者の探索精度に依存する
4. **同期方針の再判断**: 前回「差異許容」とした判断を毎回再確認している

## 期待効果

- **事前調査の効率化**: 「リストの各項目について前回コミットからの差分を確認」に集約できる
- **差分ベースの精密な反映計画**: 各ファイルの具体的な変更内容（コミットハッシュ間の diff）に基づく計画策定
- **同期方針の蓄積**: 過去の判断（直接同期 / 差分適応 / 概念移植 / 対象外）が引き継がれる
- **新規ファイルの検出**: リストにないファイルの出現を機械的に検出できる

## 補足・参考情報

- **同期対象リストの管理項目案**:
  - 対象ファイルパス（ソース側 / ターゲット側）
  - 同期方針（直接同期 / 差分適応 / 概念移植 / 対象外）
  - 前回同期時のコミットハッシュ（ソース側 / ターゲット側）
  - 前回同期日
  - 差異許容の判断理由（対象外の場合）

- **運用フロー案**:
  1. 同期セッション開始時にリストを読み込み、各項目の前回コミットからの diff を取得
  2. diff がある項目のみ反映計画に含める
  3. リストにない新規ファイルを検出し、同期方針を決定してリストに追加
  4. 同期完了時にコミットハッシュを更新

- **配置先候補**: `sessions/sync/` 直下（`sync_targets.csv` or `sync_targets.md`）または `.claude/skills/sync-manager/` 配下

- **関連施策**: 「外部リポジトリアーキテクチャオーバービュー管理」と相互補完的。オーバービューが構成全体の理解を提供し、同期対象リストが個別ファイルの追跡を担う
