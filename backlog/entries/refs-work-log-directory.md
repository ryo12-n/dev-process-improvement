## 課題・背景

現在 `refs/<リポジトリ名>/knowledge.md` は蒸留された知見の最新スナップショットとして機能しているが、各検証・作業の生データ（ログ、試行錯誤の経緯、失敗と原因特定のプロセス）は git 履歴にしか残らない。

knowledge.md への蒸留時に「どの検証でどの知見が得られたか」のトレーサビリティが失われ、将来の類似問題調査時に原因特定プロセスを再参照しづらい。

## 期待効果

- **2層構造の確立**: `knowledge.md`（蒸留された最新知見） + `work-logs/`（時系列の作業ログ）により、知見の鮮度維持と履歴追跡を両立
- **蒸留サイクルの制度化**: 作業ログ → knowledge.md への蒸留フローを明確化し、知見の取りこぼしを防止
- **類似問題の調査効率向上**: 過去の試行錯誤プロセスを作業ログから直接参照可能

## 補足・参考情報

- 具体例: `refs/claude-code-action/work-logs/20260315-gha-test-debugging.md` のような形式
- 作業ログには検証手順、GHA ログの要約、失敗原因の分析、解決策の比較検討などを記録
- knowledge.md の各知見エントリに「元ログ: work-logs/xxx.md」のリンクを付与すると逆引き可能
- 全 refs/ 配下のリポジトリ知見ディレクトリに横展開可能なパターン
