## 課題詳細

`initiative-dispatcher.yml` の Route command ステップで、ユーザーのコメント本文から抽出した `ARGS` 変数を `dispatch_inputs` の JSON 文字列に直接展開している。

具体的には:
- 行110: `"dispatch_inputs={\"issue_number\":\"${ISSUE_NUMBER}\",\"command\":\"approve\",\"feedback\":\"${ARGS}\",\"phase\":\"wallbashing\"}"` — `ARGS` がエスケープなしで JSON に展開
- 行162: `eval gh workflow run "$WORKFLOW" $input_flags` — `eval` による間接実行

`ARGS` にダブルクォート、バックスラッシュ、`$()`、バッククォート等が含まれる場合に:
1. JSON が壊れて `jq` パースが失敗し、dispatch が不発になる可能性
2. `eval` を経由するため、シェルコマンドインジェクションが成立する可能性

ISS-058 の `${{ }}` 直接展開問題と関連するが、本件はシェル変数展開 + `eval` の組み合わせでより重大度が高い。

## 対応方針（案）

[案A] `jq` による JSON エスケープ: `dispatch_inputs` を `jq -n --arg` で構築する

```bash
dispatch_inputs=$(jq -n \
  --arg issue_number "$ISSUE_NUMBER" \
  --arg command "approve" \
  --arg feedback "$ARGS" \
  --arg phase "wallbashing" \
  '{issue_number: $issue_number, command: $command, feedback: $feedback, phase: $phase}')
```

[案B] JSON 構築を廃止し、各コマンド分岐で `-f` フラグを直接構築する: case 文の各分岐で `gh workflow run` を直接実行し、JSON 中間表現と `eval` を両方排除する

[共通] `eval` の使用を禁止し、GHA ワークフロー作成ガイドラインに追加する
