## 課題詳細

openspec-code-trial の実施で、spec 作成フェーズ（proposal → specs）が最もコンテキストを消費するボトルネックであることが判明。要件の粒度（どこまで細かく書くか）と網羅性（エッジケース・境界条件の洗い出し）の判断にコンテキストが集中する。proposal での Capabilities 設計と、spec での観測可能な振る舞いへの集中が重要。

## 対応方針（案）

[案] spec 作成時の支援として、① proposal の Capabilities から spec シナリオを導出するチェックリスト・ガイドラインの整備、② spec レビュー観点（粒度・網羅性）の明文化、③ explore モードでの事前調査フェーズの活用推奨を検討する。
