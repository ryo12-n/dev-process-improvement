# 実施計画: Claude Code 標準 plugin 情報整理

## フェーズ構成

| フェーズ | 内容 | 期間 | 成功基準 |
|---------|------|------|----------|
| 1 | 公式プラグイン調査・適用可能性評価・リファレンス文書作成 | 1セッション | 評価サマリーテーブルと主要プラグインの詳細評価が完成していること |

## スケジュール

- L2-worker: プラグイン調査・評価・リファレンス文書作成
- L2-evaluator: リファレンス文書の品質評価
- L1: ゲート判定

## 成功基準（全体）

1. 公式プラグインのうち主要カテゴリ（開発フロー系、設定・自動化系、LSP系）をカバーした評価サマリーテーブルが作成されていること
2. dev-process-improvement プロジェクトへの適用可能性が「高・中・低・対象外」で分類されていること
3. `.claude/skills/plugin-reference/SKILL.md` としてリファレンス文書が配置されていること
4. `builtin-skills-reference` と同等の記述粒度（機能概要、適用可能性、具体的利用シーン、制約事項）を持つこと

## リソース・前提条件
- Web 調査が可能なこと（公式ドキュメント・GitHub・技術記事の参照）
- 壁打ちでユーザーから提供されたプラグイン情報を出発点とする
- `builtin-skills-reference/SKILL.md` のフォーマットを踏襲する

## リスクと対策
| リスク | 発生確率 | 影響 | 対策 |
|--------|---------|------|------|
| Web 調査で最新情報にアクセスできない | 中 | 中 | 壁打ちで得たユーザー情報と公式ドキュメントを組み合わせる |
| プラグイン数が多く全件評価が困難 | 中 | 低 | カテゴリ別に代表的なプラグインを評価し、全件網羅は必須としない |
