# 改善施策提案: automation-action-step

## 背景・課題
automation-manager は定期メンテナンスセッションとして自動化候補のスキャン・再評価・進捗追跡を行い、`03_report.md` に変更提案をまとめる。しかし、Step 7「アクション実施」が1行の概要記述のみで具体的なサブステップがないため、実際のセッションでは候補レジストリ（CSV/entries）の更新や backlog 起票が実行されずに PR が作成される。

さらに、ルールセクションに「候補管理の変更はユーザー承認後に実施する」という記述があり、「PR 作成前にアクションを実施する」という作業フローと矛盾している。

他の定期マネージャー（triage, metacognition, backlog-maintenance）は全て詳細な Step 7 サブステップを持ち、セッション内で変更を適用してから PR を作成する設計になっている。automation-manager だけがこのパターンから外れている。

## 目標
automation-manager の SKILL.md を修正し、他の定期マネージャーと同等の「アクション実施」ステップを具備させる。

## スコープ
### やること
- SKILL.md の Step 7 にサブステップを列挙（新規候補登録、ステータス変更、優先度変更、backlog 起票、知見ルーティング、課題 CSV 転記）
- 担当ファイルテーブルに操作対象ファイルを追加
- ルールセクションの矛盾を解消
- やることセクションの具体化
- PR サマリテンプレートを「提案アクション」→「実施済みアクション」に更新

### やらないこと
- 他の定期マネージャーへの横展開（調査の結果、triage/metacognition/backlog-maintenance は既に対応済み）
- docs/workflow.md の変更（L545-549 に既に正しい記述があり整合済み）
- テンプレートファイルの変更（03_report.md テンプレートは既に変更案セクションを持つ）

## 期待される効果
- automation-manager セッションが候補レジストリ更新・backlog 起票・知見ルーティングをセッション内で完結するようになる
- PR にレポートだけでなく実際の変更が含まれ、ユーザーが一括でレビュー・承認できる
- 他の定期マネージャーとのパターン統一による運用の一貫性向上

## リスク

|--------|--------|------|

## 壁打ちの背景
- 実装マネージャー（l1-impl-manager）の使用を検討したが、コード実装ではなくスキル定義の修正であるため通常の l1-manager を選択
- 横展開の必要性を調査した結果、automation-manager のみの問題と判明しスコープを限定
- docs/workflow.md が既に正しいため SKILL.md 側を追いつかせる方針で合意
