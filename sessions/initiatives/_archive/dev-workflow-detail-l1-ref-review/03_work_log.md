### [2026-03-15 01:20] タスクID: T-008
**状態**: 完了
**作業内容**: `gh api` アプローチ vs 2リポジトリクローン アプローチの比較評価

| 観点 | `gh api` アプローチ | 2リポジトリクローン アプローチ |
|------|-------------------|---------------------------|
| 対応環境 | GHA / SDK（1リポジトリ制約に依存しない） | CLI（ローカル環境推奨） |
| 読み取り | base64 デコードが必要、ページング考慮 | 通常のファイルシステム操作で直感的 |
| grep / 検索 | レスポンス JSON を解析する必要あり（ツール制約で実質困難） | 標準ツール（grep, ripgrep 等）が直接使用可能 |
| git 操作 | 不可（gh api は読み取りのみ） | diff, log, blame 等すべての git コマンド利用可能 |
| ファイル変更・コミット | 不可（別途 PUSH エンドポイントが必要） | 通常の git コマンドで可能 |
| `.claude/` 全体の把握 | 多数の個別リクエストが必要（ディレクトリごとにリクエスト要） | ローカルにあるため一括参照・横断検索が容易 |
| 初期設定コスト | 低（gh コマンドのみ） | 高（2リポジトリのクローン・セットアップが必要） |
| セキュリティ | READ 権限のみで十分 | WRITE 権限が必要な場合もある |
| 本施策での実績 | T-002 で1リクエスト（全文取得）成功。T-007 で .claude/ 全体把握に数十リクエスト必要 | - |

**結論**:
- 本施策のような **読み取り専用精査 × GHA 環境** の場合: `gh api` は機能的に問題なし（SDK の1リポジトリ制約を回避できる）
- ただし、**ai-driven-dev-patterns の `.claude/` 全体（スキル・ルール・エージェント）を広く把握する必要がある調査** は `gh api` では多数のリクエストが必要で非効率。CLI 環境 + 2リポジトリクローンの方が優れている
- reject フィードバックで指摘された「`.claude/` 配下全体の確認」は、今回 `gh api` で対応したが、ディレクトリ・ファイルを個別に取得する手間がかかった。クローン環境なら一括参照できる
- **推奨指針**: 対象リポジトリへの変更が不要な精査系施策でも、対象リポジトリの `.claude/` 設計文書を広く参照する可能性がある場合は、CLI 環境 + 2リポジトリクローンを検討する価値あり（→ backlog の「対象リポジトリ」列に `ai-driven-dev-patterns` が含まれる場合は CLI 推奨）

**成果物**: 本比較評価記録（Issue コメント形式での返却は `08_gate_review.md` ゲートサマリーに含める）
