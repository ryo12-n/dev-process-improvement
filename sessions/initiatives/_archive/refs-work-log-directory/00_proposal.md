# 改善施策提案: refs-work-log-directory

## 背景・課題

`refs/<リポジトリ名>/knowledge.md` は蒸留された知見の最新スナップショットとして機能しているが、各検証・作業の生データ（ログ、試行錯誤の経緯、失敗と原因特定のプロセス）は git 履歴にしか残らない。

knowledge.md への蒸留時に「どの検証でどの知見が得られたか」のトレーサビリティが失われ、将来の類似問題調査時に原因特定プロセスを再参照しづらい。

## 目標

`refs/<リポジトリ名>/` 配下に `work-logs/` ディレクトリを導入し、`knowledge.md`（蒸留済み知見）と `work-logs/`（時系列の生データ）の2層構造を確立する。

## スコープ
### やること
- `refs/_work_log_template.md` の作成（作業ログテンプレート）
- `refs/_knowledge_template.md` への `元ログ` 列追加（逆引きリンク用）
- `manager-common-policy` §5.3 の手順拡張（作業ログ作成ステップの追加）
- `CLAUDE.md` の refs/ 説明更新
- `gha-guideline/SKILL.md` への作業ログ言及追加
- 全 `refs/` 配下のリポジトリに横展開可能なパターンとして定義

### やらないこと
- 既存 `refs/claude-code-action/knowledge.md` のテーブル改修（バックフィルなし）
- 既存エントリへの遡及的な元ログリンク付与
- `l1-manager` / `l1-impl-manager` / `triage-manager` 等の間接参照ファイルの変更（§5.3 経由で反映される）

## 外部リポジトリ

| リポジトリ名 | ローカルパス | 変更内容の概要 |
|-------------|------------|-------------|
| なし | - | - |

## 期待される効果

- 2層構造の確立により、知見の鮮度維持と履歴追跡を両立
- 蒸留サイクルの制度化（作業ログ → knowledge.md）で知見の取りこぼし防止
- 類似問題調査時に過去の試行錯誤プロセスを直接参照可能

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| 作業ログ作成が形骸化する | 低 | 作業ログは任意（小規模知見は不要）とし、運用負荷を最小化 |

## 壁打ちの背景

- 対象範囲: 全 `refs/` 配下に横展開（パターン定義 + テンプレート提供で対応）
- 既存エントリの改修: 新規エントリからの適用とし、バックフィルは行わない
- 作業ログの粒度: 1セッション = 1ファイル（`YYYYMMDD-<topic>.md` 形式）
- 作業ログは任意ステップとし、小規模知見は直接 knowledge.md に追記可能

## 備考・設計パターン

作業ログ命名規則: `refs/<repo>/work-logs/YYYYMMDD-<topic>.md`
- `YYYYMMDD`: 調査セッションの日付
- `<topic>`: ケバブケースの調査トピック名（例: `gha-permission-debugging`）
