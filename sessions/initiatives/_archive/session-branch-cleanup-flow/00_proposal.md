# 改善施策提案: セッション開始時ブランチ整理フロー標準化

## 背景・課題

- セッション起動時に前回作業のブランチが残っていることがある（未プッシュコミット・PR未作成の状態）
- この状態で新しい施策やトリアージを開始すると、作業が混在するリスクがある
- 現在の L1マネージャー・トリアージマネージャーのスキル定義にはセッション開始前のブランチ状態チェックが含まれていない
- 発生契機: README修正作業時に前セッションのブランチが残っていた実体験

## 目標

- セッション開始時にブランチ整理が確実に行われるフローを標準化する
- `.claude/rules/` に共通ルールとして配置し、L1・トリアージの両セッションで参照する

## スコープ

### やること

- `.claude/rules/session-start-branch-cleanup.md` を新規作成（共通ルール）
- `.claude/skills/l1-manager/SKILL.md` にルール参照を追加
- `.claude/skills/triage-manager/SKILL.md` にルール参照を追加
- `docs/workflow.md` に連動更新

### やらないこと

- L2サブエージェント（worker / evaluator）への適用（サブエージェントはL1から起動されるため不要）
- ブランチ自動マージの実装（マージはユーザー判断に委ねる）

## 期待される効果

- セッション開始時に前回作業の整理が確実に行われる
- 未プッシュ・未PR状態の作業が放置されることを防止
- 作業混在によるコンフリクトや情報散逸のリスクを低減

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| フロー追加によるセッション開始の遅延 | 低 | ステップは軽量（git status + push + PR作成程度） |
| mainブランチが最新でない場合のコンフリクト | 低 | pull --rebase で最新化する手順を含める |

## 壁打ちの背景

- **フロー発動条件**: 「mainでなければ整理を実施」ではなく、常にフローを発動する方針に決定。同じ施策の続きでも一旦mainに戻してから再度ブランチに入る
- **マージ待ちの扱い**: PR作成済みを確認したら先に進んでよい方針に決定。マージ待ちでセッションをブロックしない
- **適用範囲**: L1・トリアージのみ。L2はサブエージェントとして起動されるため対象外
- **ルール配置**: `.claude/rules/` に共通ルールとして切り出す。適用対象はL1・トリアージに限定されるが、セッション開始時の共通動作であるため共通ルール化が妥当と判断

## 備考・設計パターン

- ルールファイルのフロー案（バックログから引用・調整）:
  1. 現在のブランチとリモート未反映のコミットを確認
  2. 未プッシュコミットがあればプッシュ
  3. main向けPRが未作成であれば作成
  4. PR作成済みを確認したらmainブランチに切り替え・最新化
  5. 通常のセッションフローへ進む
