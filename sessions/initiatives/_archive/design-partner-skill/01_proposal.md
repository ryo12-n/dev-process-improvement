# 改善施策提案: design-partner-skill

## 背景・課題

現在のL1マネージャーの壁打ちフェーズは「施策が既にある程度定まっている」前提で動作する。しかし実際には、ユーザーが「何をやりたいか自体がまだ曖昧」な段階からの伴走が必要なケースが多い。

現状の問題点:
- 曖昧なアイデアを構造化する仕組みがない（inbox に書いてトリアージを待つか、直接 L1 に投げるしかない）
- L1 の壁打ちフェーズは施策の方向性確認が主目的であり、アイデアの発散・収束を支援する設計ではない
- 壁打ちの過程（思考の変遷・却下した選択肢）が記録として残らない

## 目標

「壁打ち → 具体化 → 設計 → 実施計画」を一貫して伴走するスキルを作成し、曖昧なアイデアから backlog エントリ + 設計成果物を生成できるようにする。

## スコープ
### やること
- design-partner スキル（SKILL.md）の作成
- 対話エンジン（壁打ち層）: 発散・制約明確化・収束の質問パターン設計
- 構造化エンジン（設計層）: 要件定義・WHY/WHAT/HOW分解・スコープ境界テンプレート
- 実施計画生成（アクション層）: タスク分解・依存関係マッピング
- 成果物の置き場（`sessions/design/`）のディレクトリ構成定義
- backlog エントリへのポインター連携

### やらないこと
- L1 マネージャーの壁打ちフェーズの変更（本スキルは L1 の上流に位置する独立スキル）
- ワーカー/評価者エージェントの作成（本スキルは人間との直接対話で完結する）
- 汎用化（dev-process-improvement リポジトリ外への適用）

## 期待される効果

- アイデアの具体化にかかる認知コストの低減
- 壁打ち過程の構造化された記録が残る
- backlog エントリの品質向上（設計成果物へのポインター付き）
- L1 マネージャーが拾う時点で proposal/plan の素案が既にある状態を作れる

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| L1の壁打ちフェーズとの責務重複 | 中 | 本スキルは「アイデア→backlog」、L1は「backlog→施策」と明確に分離。本スキルは proposal/plan を作らない |
| フェーズ管理が形骸化する | 低 | 各フェーズにユーザー承認ゲートを設ける |
| 成果物が使われない | 中 | backlog エントリに設計成果物へのポインターを記載し、L1 が参照する導線を作る |

## 壁打ちの背景

ユーザーとの壁打ちで以下の方針が決定された:

1. **成果物の置き場**: `sessions/design/` 配下に専用フォルダを作成する（案A採用）
   - inbox 経由やbacklog直接作成ではなく、壁打ちログ・設計ドキュメント・実施計画を構造化して保存する場所が必要
2. **既存フローとの境界**: backlog エントリ作成までがスコープ（案A採用）
   - L1 が拾って proposal/plan/tasks を作る。本スキルはL1のフェーズAを代替しない
3. **起動方法**: 人間が直接 `/design-partner` で呼び出す
   - L1 マネージャーの拡張ではなく、独立したスキル
4. **対話スタイル**: 質問をまとめて提示 → 回答を受けて次フェーズへ
5. **人間の関与**: AIが仮説提示 + 選択肢提示の両方を使い分ける

### 参考情報（ユーザー提供の設計素案）

3レイヤー構成:
- **対話エンジン（壁打ち層）**: 目的発散 → 制約明確化 → アイデア収束（MoSCoW等）
- **構造化エンジン（設計層）**: WHY/WHAT/HOW分解、スコープ境界明示、レビューチェックポイント
- **実施計画生成（アクション層）**: タスク分解ルール、依存関係マッピング

フェーズ管理: EXPLORE → DEFINE → DESIGN → PLAN → REVIEW（ユーザー承認で遷移）

## 備考・設計パターン

本スキルはマネージャー＋ワーカー構成ではなく、**単体スキル**として実装する。人間との直接対話で完結するため、サブエージェントは不要。session-flow-policy の §1（マネージャー＝スキル、ワーカー＝エージェント）の構造は適用しない。
