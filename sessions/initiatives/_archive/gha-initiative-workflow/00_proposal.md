# 改善施策提案: GHA Initiative ワークフロー

## 背景・課題

現在、initiative セッション（L1/L2 パターン）は `.claude/skills/l1-manager/` で CLI/SDK 環境向けに実装されている。GHA 環境では `backlog-auto-execute.yml` が非対話で全フェーズを一括実行するが、以下の課題がある:

- 壁打ちフェーズがない（人間のフィードバックを反映する機会がない）
- ゲート判定の承認ステップがない（自動で完了してしまう）
- Issue を介した人間との対話がない

skills 版の initiative セッションでは壁打ちとゲート判定の2箇所で人間の承認を求めるが、GHA 版にはこの仕組みがない。

## 目標

GitHub Actions ワークフローとして initiative セッションを実装し、GitHub Issue を介して人間と壁打ち・ゲート判定の対話を行える仕組みを構築する。

## スコープ
### やること
- initiative 用の GHA ワークフロー3ファイルの作成（wallbash / execute / close）
- ラベル設計（initiative 固有の名前）
- Issue コメントによる `/approve` `/reject` フロー実装
- gha-guideline への権限マトリクス追記

### やらないこと
- 既存の `backlog-auto-execute.yml` の変更（対話版として共存）
- プロンプトファイルの外部分離（今後の施策で検討）
- E2E テストの自動化

## 期待される効果

- GHA 環境でも人間のフィードバックを反映した施策実行が可能になる
- Issue を通じた非同期の壁打ち・承認フローが実現する
- 施策の進行状態がラベルで可視化される

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| issue_comment トリガーが bot のコメントでも発火する | 中 | `github.event.comment.user.login != 'github-actions[bot]'` でフィルタ |
| フェーズ間の状態引き継ぎが失敗する | 中 | PR ブランチを軸にし、Issue 番号から PR を検索する方式で一貫性を確保 |
| Claude の max-turns が不足する | 低 | execute フェーズは 80 turns に設定 |

## 壁打ちの背景

- ワークフロー分割はフェーズごと（3ファイル）に決定。壁打ちフェーズで PR を作成し、同じブランチで後続フェーズが作業を継続する方式
- 承認方法はコメント方式（`/approve [フィードバック]`）を選択。ラベル方式より人間のフィードバック内容を Claude が直接読めるメリットがある
- トリガーは `issues:labeled` + `workflow_dispatch` + `issue_comment` の3種を組み合わせ
- 入力ソースは Issue 本文と backlog エントリの両方に対応

## 備考・設計パターン

本施策は既存の `backlog-auto-execute.yml`（非対話型）の対話版として共存する位置づけ。`gha-guideline` の3層権限モデルに準拠する。
