# 改善施策提案: GHA 壁打ちフェーズの独立・反復化

## 背景・課題

現状の GHA Initiative フローでは壁打ちが1回きりの単発実行（`/start` → 壁打ち → `/approve` → 実行）であり、以下の問題がある:

1. **壁打ちの反復手段がない**: `/start` で1回壁打ちが走った後、フィードバックは `/approve <feedback>` で execute フェーズに渡すしかない。壁打ちフェーズ内での反復ができない
2. **コンテキストが断絶する**: 各 GHA run は新規 Claude セッションを起動するため、前回の調査・推論が失われる。フィードバックを踏まえた一貫性のある回答ができない
3. **壁打ちと proposal 作成が混在**: 壁打ちフェーズで `00_proposal.md` を作成するが、計画（`01_plan.md`）やタスク（`02_tasks.md`）は execute フェーズで初めて作成される。壁打ち段階で計画の方向性を議論・合意できない

## 目標

1. 壁打ちを独立した反復可能フェーズとして分離する
2. 人間が承認するまで何度でも壁打ちを繰り返せるようにする
3. AI が調査ログ・判断材料をファイルとして保持し、反復間でコンテキストを継続する
4. 壁打ち完了時に proposal/plan/tasks のドラフトが揃っている状態にする

## スコープ

### やること
- `00a_wallbash_log.md` テンプレートの新規作成（コンテキスト保持用）
- `02_tasks.md` テンプレートにディスパッチ計画セクションを追加
- `gha-wallbash/SKILL.md` を init/continue の2モード対応に書き換え
- `initiative-wallbash.yml` に新 inputs（mode, feedback, branch, initiative_dir）追加 + PR 作成削除
- `initiative-dispatcher.yml` に `/continue` コマンドを追加
- `initiative-execute.yml` に PR 作成ステップを追加（壁打ちフェーズからの遷移時）
- `gha-execute/SKILL.md` を壁打ちドラフト活用に対応
- `l1-manager/SKILL.md` に `00a_wallbash_log.md` 記録ステップを追加（CLI アライメント）
- `docs/workflow.md` の更新

### やらないこと
- CLI 側の壁打ちフローの大幅な変更（既に対話的にマルチターン可能なため）
- ラベル体系の変更（既存の `initiative:wallbashing` をそのまま使用）
- execute/close フェーズの大幅な変更

## 期待される効果

- 壁打ちの品質向上: 反復により AI の提案精度が上がり、execute フェーズでの手戻りが減少
- コンテキスト継続: 調査ログにより一貫性のある回答が可能に
- 計画の早期合意: 壁打ち段階で plan/tasks のドラフトが見えることで、方向性のずれを早期に検出

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 壁打ちログの肥大化でコンテキストウィンドウを圧迫 | 中 | 「現在の合意事項」セクションを毎回上書きし、効率的に読めるようにする |
| 後方互換の破壊（既存の wallbashing フェーズの施策） | 中 | execute スキルにフォールバック処理を追加（ドラフトなし時は従来動作） |
| ワークフロー inputs の追加による dispatcher の複雑化 | 低 | `/continue` コマンドの routing は `/approve` と同構造で実装 |

## 壁打ちの背景

- 壁打ちの反復化は GHA フローの根本的な UX 改善。現状は「1回で提案を作り、approve で即実行」だが、人間が AI の提案を吟味・修正するフローが欠如していた
- PR 作成は壁打ち承認時（`/approve`）に行う設計とした。壁打ち中は Issue コメントでやり取りし、PR レビューは成果物確定後に行う
- コンテキスト保持ファイルの番号は `00a` とした。proposal (00) の前段であることを示し、既存の 00-08 番号体系を崩さない
- ディスパッチ計画は `02_tasks.md` のセクションとして追加。別ファイルにすると番号体系やスキル参照の変更が大きくなるため
- CLI 側は既に対話的に壁打ち可能なため、`00a_wallbash_log.md` の記録追加のみで整合を取る

## 備考・設計パターン

新フロー:
```
/start → wallbash (初回) → [Issue コメントでレビュー]
  → /continue <feedback> → wallbash (反復 N) → [レビュー]
  → /approve → PR 作成 → execute → close
```
