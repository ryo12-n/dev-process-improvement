# 改善施策提案: l1-impl-manager セッションタイプの新設

## 背景・課題

現在の `l1-manager` セッションはプロセス改善施策（ドキュメント・ルール変更）に特化している。コード実装タスクを扱う場合、以下の課題がある:

1. **フェーズ粒度の不一致**: 現行は「実施→評価→ゲート」の1サイクル構成だが、コード実装は「調査→設計→実装計画→実装」の4フェーズが必要
2. **並列実装の未対応**: 実装フェーズでは複数ファイルを並列ワーカーで分担したいが、現行の逐次ディスパッチでは対応できない
3. **作業ログの欠如**: L2ワーカー（サブエージェント）は歴史的にwork_logを書かない問題がある。コード実装ではテスト失敗・リトライの経緯が重要
4. **外部リポジトリとの連携**: コードは外部リポジトリに存在し、管理ドキュメントは dev-process-improvement に置く二重管理が前提

## 目標

- コード実装専用の4フェーズマネージャーセッション `l1-impl-manager` を新設する
- 各フェーズにワーカー＋評価者のペアを持ち、4回のゲート判定で品質を担保する
- 実装フェーズでファイルレベルの並列ワーカーディスパッチを実現する
- ワーカーの作業ログを「チェックポイント記録(C) + マネージャーによるgit履歴集約(B)」のB+C方式で確保する

## スコープ

### やること

1. `l1-impl-manager` の SKILL.md を作成（入力パターン・ライフサイクルtodo・4フェーズオーケストレーション）
2. 5つのエージェント定義を作成:
   - `investigation-worker.md` — Phase 1: コードベース調査
   - `design-worker.md` — Phase 2: 設計ドキュメント作成
   - `impl-plan-worker.md` — Phase 3: ファイルレベルタスク分割
   - `impl-worker.md` — Phase 4: コード実装（テストリトライ・エスカレーション付き）
   - `impl-evaluator.md` — 全フェーズ共通評価者（フェーズパラメータで動作切替）
3. `sessions/impl/_template/` に4フェーズ対応テンプレートを作成
4. 既存ファイルの連動更新（commit-message.md、docs/workflow.md、triage-standard-policy 等）

### やらないこと

- 既存 `l1-manager` の変更（別セッションタイプとして共存）
- 大規模リファクタリング対応（スコープは1ブランチ規模）
- SDK環境対応（ISS-027制約によりローカルCLI専用）

## 期待される効果

- コード実装タスクに特化したフレームワークにより、調査→設計→計画→実装の各フェーズで品質を段階的に担保
- ファイルレベルのタスク分割による並列実装でスループット向上
- B+C方式によるワーカー作業ログの自動的な確保

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| テンプレートファイル数の増加（17+ファイル）による管理コスト | 中 | フェーズプレフィックス番号（10/20/30/40）で整理、不要フェーズはスキップ可能な設計 |
| 評価者1エージェントで4フェーズを汎用的にカバーできるか | 低 | マネージャーからのフェーズ固有コンテキスト注入で対応 |
| 並列ワーカー間のファイル競合 | 高 | Phase 3 で Conflict Check Matrix による事前検証を義務化 |

## 壁打ちの背景

### 確定した設計判断

1. **4フェーズ構成**: Investigation → Design → Implementation Planning → Implementation
2. **各フェーズにワーカー＋評価者ペア**: 4回のゲート判定
3. **フェーズ専用エージェント**: 各フェーズに専用ワーカー、評価者は1つを共有
4. **並列実装ワーカー**: Phase 3 のファイルレベルタスク分割で競合を事前排除
5. **B+C方式の作業ログ**:
   - B: マネージャーがワーカー完了後にgitコミット履歴からwork_reportを構築
   - C: ワーカーがタスク開始/完了時にチェックポイントを構造化記録
6. **スコープ**: 1ブランチ規模（単一機能/修正）
7. **別セッションタイプ**: l1-manager のモード切替ではなく独立セッション
8. **ローカルCLI専用**: ISS-027（SDK環境の1リポジトリ制約）

### ファイル番号体系

フェーズプレフィックス方式を採用:
- `00-08`: グローバルファイル（proposal, plan, issues, gate_summary）
- `10-15`: Phase 1 (Investigation)
- `20-25`: Phase 2 (Design) + `02_impl_design.md`
- `30-36`: Phase 3 (Implementation Planning) + `36_file_task_division.md`
- `40-45`: Phase 4 (Implementation) + 動的ワーカーログ

### エージェント構成

- ワーカー4種（フェーズ専用）+ 評価者1種（共通）= 5エージェント
- 評価者はフェーズ固有コンテキストをマネージャーから受け取る

## 備考・設計パターン

- **sessions/impl/**: `sessions/initiatives/` とは別ディレクトリで管理（コード実装 vs プロセス改善の区分）
- **Phase 3 → Phase 4 のハンドオフ**: `36_file_task_division.md` の Conflict Check Matrix で並列安全性を検証してからディスパッチ
- **テストリトライループ**: impl-worker に max 3回のリトライ、超過時はエスカレーション
- **エスカレーション条件**: 設計矛盾、スコープ外ファイル変更、テスト3回失敗
