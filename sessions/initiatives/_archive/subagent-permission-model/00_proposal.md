# 改善施策提案: サブエージェントの権限許可モデル調査

## 背景・課題

L2サブエージェント（Agent ツール）内で実行される Bash コマンドが、`.claude/settings.json` の allow リストで許可済みにも関わらずユーザーに許可を求めていた可能性がある。`session-branch-cleanup-flow` 施策の L2 サブエージェント実行時に発生を観測。

サブエージェントの権限継承モデルについて正確な理解がないまま運用しており、設定の最適化や運用ガイドが存在しない状態。

## 目標

1. サブエージェントの権限継承モデルを公式ドキュメント調査 + 実機検証で明らかにする
2. 検証結果に基づく運用ガイドを `docs/` に作成する
3. `.claude/settings.json` およびエージェント定義の改善提案を行う

## スコープ
### やること
- 公式ドキュメント調査（完了済み。結果は壁打ちの背景に記載）
- allow/deny/非allow コマンドの実機検証
- エージェント定義の tools フィールドと権限の関係検証
- バックグラウンド vs フォアグラウンド実行時の権限制約調査
- 運用ガイド `docs/subagent-permission-guide.md` の作成
- settings.json・エージェント定義の改善提案（work_report に記載）

### やらないこと
- settings.json やエージェント定義の直接変更（L2 は提案のみ、L1 が判断・実施）
- Claude Agent SDK レベルの詳細調査（本PJ は Claude Code CLI の範囲）
- 他リポジトリの設定変更

## 期待される効果

- L2 サブエージェントの権限モデルの正確な理解により、不要な許可プロンプトの原因特定と解消
- 運用ガイドにより今後の設定変更時の判断基準が明確化
- エージェント定義の最適化提案

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| 実機検証で再現できない（許可プロンプトが発生しない） | 低 | 検証結果自体が「正常動作確認」として価値がある |
| バックグラウンド実行の検証が困難 | 中 | ドキュメント調査結果で代替し、課題として起票 |

## 壁打ちの背景

### 公式ドキュメント調査結果（フェーズ0で完了）

claude-code-guide エージェントによる調査で以下が判明:

1. **サブエージェントは親セッションの allow/deny リストを継承する**（公式ドキュメント: "Each inherits the parent conversation's permissions with additional tool restrictions."）
2. **バックグラウンド実行時の制約**: 起動前にプリアプルーブが必要。プリアプルーブ外の操作は自動拒否
3. **Permission 評価順序**: Hooks → Deny rules → Permission mode → Allow rules → canUseTool callback
4. **エージェント定義での制限**: `tools` フィールド（allowlist）と `disallowedTools` フィールド（denylist）で追加制限可能
5. **`permissionMode` の継承**: 親が `bypassPermissions` なら子も継承。ただし `bypassPermissions` では `allowedTools` は効果なし（全ツール承認）

### スコープ合意
- 検証まで行いたい（ドキュメント調査だけでなく実機検証を含める）
- settings.json の修正や運用ガイドの作成まで含める

## 備考・設計パターン

検証アプローチ: L2-worker 自身がサブエージェントとして動作しているため、自身の環境で Bash コマンドを実行し許可プロンプトの発生有無を観察することで検証データとする。
