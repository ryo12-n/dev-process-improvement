# 実施計画: GHA トリアージ定期実行

## フェーズ構成

| フェーズ | 内容 | 成功基準 |
|---------|------|----------|
| 1 | GHA ワークフロー作成・プロンプト設計 | ワークフローファイルが構文的に正しく、プロンプトがトリアージフローを網羅している |

## 成功基準（全体）

1. `.github/workflows/daily-triage.yml` が作成され、schedule + workflow_dispatch トリガーが設定されている
2. トリアージ用プロンプトが非対話モードで完全なトリアージフロー（事前調査 → ワーカー → 評価者 → レポート → ゲート判定）を実行できる設計になっている
3. 空振り対策（inbox 0件時の早期終了）が組み込まれている
4. ブランチ戦略（`triage/YYYYMMDD`）と PR 作成が設計されている
5. セキュリティ: API キーが secrets 経由で安全に管理されている

## リソース・前提条件

- GitHub リポジトリに `ANTHROPIC_API_KEY` シークレットが設定されていること（マージ後に人間が設定）
- GHA ランナーで Claude Code CLI が利用可能であること（npm install or curl）

## リスクと対策

| リスク | 発生確率 | 影響 | 対策 |
|--------|---------|------|------|
| GHA 環境での Claude Code サブエージェント動作不良 | 中 | 高 | プロンプト設計で単一エージェント実行にフォールバック可能な構成にする |
| API コスト超過 | 低 | 中 | `--max-turns` でターン数制限、空振り対策で不要実行を削減 |
| schedule 実行の遅延（GitHub の cron は正確でない） | 低 | 低 | 日次実行なので多少のずれは許容 |
