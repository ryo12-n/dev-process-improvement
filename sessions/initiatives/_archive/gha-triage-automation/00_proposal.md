# 改善施策提案: GHA トリアージ定期実行

## 背景・課題

トリアージセッションは現在、人間が手動で `/triage-manager` を呼び出して実行している。以下の課題がある:

1. **実行忘れのリスク**: inbox に新しいエントリが溜まっても、手動で気づいてトリアージを起動する必要がある
2. **実行頻度のばらつき**: 忙しい時期はトリアージが後回しになり、inbox が肥大化する
3. **起動コストの高さ**: 毎回 Claude Code セッションを立ち上げ、`/triage-manager` を手動で呼ぶ必要がある

## 目標

GitHub Actions の schedule トリガーで毎日トリアージセッションを自動実行し、結果を PR として提出する仕組みを構築する。

## スコープ

### やること
- `.github/workflows/daily-triage.yml` の作成（schedule + workflow_dispatch トリガー）
- トリアージ用プロンプトの設計（非対話モード向け）
- 空振り対策（inbox 0件時の早期終了）
- ブランチ戦略の設計（`triage/YYYYMMDD` ブランチ → PR）

### やらないこと
- PR の自動マージ（人間が確認してマージする）
- トリアージ以外のセッション（メタ認知、バックログメンテナンス等）の自動化（将来の別施策）
- Claude Code のインタラクティブ機能（壁打ち等）の自動化

## 期待される効果

1. トリアージの実行頻度が毎日に安定化する
2. inbox の滞留期間が最大1日に短縮される
3. 手動での起動作業が不要になる

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| API コスト増加（毎日実行） | 中 | 空振り対策で inbox 0件時は早期終了。実行頻度は cron で調整可能 |
| GHA 実行環境での Claude Code の制約 | 中 | `--dangerously-skip-permissions` の使用、事前の workflow_dispatch テスト |
| プロンプト設計の品質 | 低 | `/triage-manager` の SKILL.md を基に設計。段階的に改善 |

## 壁打ちの背景

- **実行方式**: `anthropics/claude-code-action` と CLI 直接実行の2案を検討。ユーザーの希望により **Claude Code CLI 直接実行** を採用
- **実行頻度**: 毎日・週1・柔軟の3案から **毎日** を選択。inbox の蓄積を最小限に抑える方針
- **自動化範囲**: 自動マージまで・PR作成まで・検討中の3案から **PR 作成まで** を選択。人間のレビューを挟む安全策

### 技術調査結果
- `anthropics/claude-code-action` は schedule トリガーをサポートしている
- CLI `-p` モードで非対話実行が可能
- スキル定義は `--system-prompt-file` / `--append-system-prompt` で注入可能
- `--dangerously-skip-permissions` で権限確認をスキップ可能（GHA 向け）

## 備考・設計パターン

本施策は「新規ツール導入」に該当するため、以下の2フェーズパターンを推奨:
1. **モック検証**: 簡単なプロンプト（echo 相当）で GHA ワークフロー自体の動作を確認
2. **本番検証**: トリアージプロンプトを投入して実機能を検証

ただし、本施策では GHA ワークフローファイルと プロンプトの作成が主であり、`workflow_dispatch` での手動テストが検証手段となる。実際のテスト実行はリポジトリの GHA 環境が必要なため、L2 の成果物は「ファイル作成」までとし、実環境テストはマージ後に人間が行う。
