# 改善施策提案: AI Dev Chain Bootstrap

## 背景・課題

現状、アプリケーション開発を始める際に「要望書 → 要件定義 → 設計 → テスト → 実装」の各フェーズを人間が手動でつなぐ必要があり、フェーズ間の成果物の受け渡しや構造化に多くの手間がかかっている。

Claude Code の Skills 機能（`/スキル名` コマンド）を活用すれば、各フェーズを AI が担う成果物チェーンとして自動化できる。まずはその土台となるディレクトリ構成・テンプレート・Skill 骨格を整備する。

## 目標

- `projects/_template/` を用意し、`/project-init` コマンド1つで新プロジェクトを開始できる状態にする
- `gen-req` / `gen-design` / `gen-tests` / `gen-ci` / `gen-tasks` の 6 Skills の骨格ファイルを作成し、`/スキル名` で呼び出せる状態にする
- 以降のイニシアティブで各 Skill の中身を順次実装できる土台を整える

## スコープ

### やること
- `projects/_template/` ディレクトリと各成果物プレースホルダーの作成
- 6 Skills（`project-init` / `gen-req` / `gen-design` / `gen-tests` / `gen-ci` / `gen-tasks`）の `SKILL.md` 骨格作成
- `refs/ai-dev-chain/overview.md` への全体計画書配置（完了済み）

### やらないこと
- 各 Skill のプロンプト実装（中身の作り込みは後続イニシアティブで実施）
- 実際の LLM 呼び出しロジックの実装
- サンプル要望書を使った E2E 動作確認（M8 以降）

## 期待される効果

- 次セッションから「`/gen-req` の中身を実装する」という単位で作業を進められる
- 全 Skill が `/スキル名` で呼び出し可能な骨格になることで、動作確認しながら段階的にブラッシュアップできる

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 既存 Skill 命名との衝突 | 低 | `gen-` プレフィックスで既存と区別 |
| `projects/` と `sessions/` の用途が混同される | 低 | 両ディレクトリの README に用途を明記 |

## 壁打ちの背景

- CLI ツール（独立バイナリ）ではなく Claude Code Skills として実装することを確認
- 成果物は `projects/<project-name>/artifacts/` に保存（dev-process-improvement 内のアプリ開発プロジェクトを対象）
- Skills の呼び出しは `/gen-req projects/my-api` 形式を基本とし、引数なしの場合はユーザーへ質問
- 全体計画書は `refs/ai-dev-chain/overview.md` に恒久配置し、セッション横断で参照
- 今回のイニシアティブはあくまで「土台作り」に絞り、Skill の中身の実装は後続に分割

## 備考・設計パターン

各 Skill は段階的にブラッシュアップする想定のため、初期の `SKILL.md` は最小限の構造（フロントマター + 基本フロー + TODO コメント）で作成する。
