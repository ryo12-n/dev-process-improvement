# 改善施策提案: l2-knowledge-action-gap

## 背景・課題

L2-worker が走査・作業中に知見（ルール化候補）を発見し `03_worker_report.md` に記録しても、実際のルールファイル修正に反映されないケースが発生している。

**具体的な事例**: external-repo-branch-pr-rules 評価で「`gh -C` 互換性を検証すべき」という知見が発見されたが、ルールファイルは `gh -C` のまま残存した。

**根本原因**: 現行フローでは知見の「記録」は義務化されているが、「記録した知見に基づくアクション（修正適用）」のステップが欠落している。L2 は知見をレポートに書くだけで作業完了となり、L1 のゲート判定でも知見のルーティング先（inbox/backlog/refs 等）は決めるが、「即時適用可能なルール化候補が修正済みか」の検証は行わない。

## 目標

- L2 が発見した「ルール化候補」知見のうち、即時適用可能なものが同一施策内で修正適用される率を向上させる
- 知見発見→修正適用のフィードバックループを L2 作業フロー + L1 ゲート判定の二重防御で確立する

## スコープ
### やること
- L2-worker エージェント定義に「知見に基づく修正適用」ステップを追加
- L2-evaluator エージェント定義に「知見に基づく修正提案」ステップを追加
- L1 ゲート判定（05_gate_review.md テンプレート）に「ルール化候補の修正適用ステータス確認」観点を追加
- L1 マネージャー SKILL.md の成果物確認観点に修正適用チェックを追加

### やらないこと
- 知見記録の仕組み自体の大幅変更（現行の「ルール化候補」「参考情報」分類は維持）
- ワーカーレポート・評価レポートのテンプレート構造変更
- manager-common-policy の知見ルーティングフロー（§5）の変更

|-------------|------------|-------------|
| なし | - | - |

## 期待される効果

- L2 が発見した即時適用可能な知見の修正漏れがゼロに近づく
- ゲート判定時に「知見は記録されたがアクション未実施」の状態が検出・対処される
- 知見の価値が施策内で即時回収され、次施策への先送りが減る

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| L2 のスコープが広がり作業時間が増加する | 中 | 「即時適用可能」の判断基準を明確化し、スコープ外のものは従来通りルーティングに回す |
| L2 が知見修正で本来のタスクスコープを逸脱する | 中 | 修正対象を「現施策で変更中のファイル」に限定する条件を明記 |

## 壁打ちの背景

- L2-worker と L2-evaluator の両方を対象とする（二重防御）
- L2 側で修正適用ステップ追加 + L1 側で修正漏れチェック追加のアプローチを採用
- 知見分類（ルール化候補 / 参考情報）は現行のまま維持し、「ルール化候補」の中で即時適用可能なものにフォーカスする

## 備考・設計パターン

知見の「即時適用可能」判断基準:
1. 修正対象ファイルが現施策のスコープ内にある
2. 修正内容が明確で、追加調査を必要としない
3. 修正が他の施策と競合しない

上記を満たさない場合は従来通り L1 のルーティング（inbox/backlog/refs）に回す。
