## 発見された課題・改善提案

### 課題1: T-001 の行番号不正確（軽微）

T-001 の Assigned files (MODIFY) 記述で以下の行番号が実ファイルと乖離している。

- **"L4-15: inputs ブロック"**: 実際は L6-30。inputs パラメータは issue_number から branch まで6つあり、retry_count の追加位置は L30 付近（branch の後）が適切。
- **"L159-179 付近（Extract gate review summary ステップ後）"**: L159-179 は Extract gate review summary ステップ自体。「後」の追加位置は L180 以降。

T-002, T-003, T-005 の行番号は正確であり、実装に支障のない範囲の誤差。ワーカーが実ファイルを Read すれば修正可能。

### 課題2: T-001 の permissions AC の表現改善（軽微）

T-001 の AC に「`permissions` が既存の最小権限を維持している（actions: write の追加が必要）」とあるが、「維持」と「追加」が矛盾する表現。推奨: 「既存の permissions に `actions: write` を追加し、それ以外の権限は変更しない」に修正。

### 課題3: T-004→T-001 の論理的依存の明示不足（軽微）

T-004（batch-approve）は `retry_count=0` を付与して initiative-execute.yml をディスパッチする。しかし `retry_count` 入力は T-001 が追加するものであり、T-001 未実装の状態で T-004 が実行されると `retry_count` パラメータは無視される（`gh workflow run` は未定義パラメータを無視する）。

**影響度**: 低。Wave 1 で全並列実行し、マージは Wave 完了後に一括で行われるため、実行時には問題にならない。ただし、Parallelism Analysis に T-001/T-005 と同様の論理的依存として記述があるべき。
