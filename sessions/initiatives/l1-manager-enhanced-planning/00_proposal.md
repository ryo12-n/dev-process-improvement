# 改善施策提案: l1-manager-enhanced-planning

## 背景・課題

L1マネージャーの施策実行能力に以下の課題がある:

1. **コンテキスト溢れ**: マネージャー自身が全計画（タスク分割・依存分析・コンフリクトチェック）を作成するため、コンテキストウィンドウが圧迫される
2. **逐次ディスパッチのみ**: ワーカーセットが1つずつの逐次ディスパッチに限定されており、独立タスクの並列実行ができない
3. **ドメインエージェント未活用**: Agency エージェント（141体）が Worker にのみ注入され、Evaluator には注入されていない。有効性のフィードバックループも存在しない
4. **フェーズ分割が固定的**: Phase A（計画）→ Phase B（実施）の2段階で、計画フェーズの品質保証が不十分

## 目標

1. 計画ワーカー（l2-plan-worker）を導入し、L1のコンテキスト負荷を軽減する
2. 並列ディスパッチ機構を実現し、独立タスクセットの同時実行を可能にする
3. Evaluator にもドメインエージェントを注入し、フィードバックループを構築する
4. フェーズ構造を Phase 1〜6 の連番方式に再構築する

## スコープ
### やること
- 計画ワーカー（l2-plan-worker）エージェント定義の新規作成
- 計画評価者（l2-plan-evaluator）エージェント定義の新規作成
- l1-manager SKILL.md のフェーズ構造再構築（Phase 1〜6）

## 期待される効果

- L1マネージャーのコンテキスト使用量が計画フェーズで 30-50% 削減
- 独立タスクセットの並列ディスパッチにより施策完了までの待ち時間短縮
- Evaluator のドメイン知識向上による評価品質の改善
- フィードバックループによるドメインエージェント推奨精度の継続的向上

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| 計画ワーカーが低品質な出力を出す | 中 | Phase 2→3 ゲートで必ず検証。差し戻しフローを定義 |
| 並列ディスパッチでファイル衝突 | 高 | コンフリクトマトリクスによる事前検証 + L1の承認ゲート |
| ドメインエージェントがノイズになる | 低 | 有効性フィードバックで追跡し、不要なら施策単位で除外可能 |
| 全施策必須で小規模施策のオーバーヘッド増 | 中 | 計画ワーカーを軽量に設計（小規模施策では最小限の出力で済むようにする） |

## 壁打ちの背景

壁打ちで以下の意思決定を行った:

- **施策分割**: 3つの独立施策ではなく、1施策・3フェーズ（1ディレクトリ）で管理する。理由: 計画ワーカー→並列ディスパッチ→ドメインエージェント統合は段階的に積み重なる変更であり、分離すると依存管理が煩雑になるため
- **Phase 2（計画ワーカー）**: 全施策で必須とする。小規模施策でもオーバーヘッドを最小限に抑える設計にする
- **計画評価者**: 専用エージェント（l2-plan-evaluator.md）を新設する。既存の l2-evaluator とは異なり、コンフリクトマトリクス・依存グラフ・Wave割当の正確性を評価する
- **ドメインエージェント注入**: Worker/Evaluator 別に選定する。同じエージェントでも役割に応じて異なるコンテキストで注入する
- **フェーズ表記**: 連番方式（Phase 1, 2, 3, 4, 5, 6）を採用。Phase A', A'' などのプライム表記は避ける
