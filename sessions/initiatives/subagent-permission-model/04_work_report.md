# 作業レポート: サブエージェントの権限許可モデル調査

## サマリ

L2-worker 自身がサブエージェントとして動作している環境で、settings.json の allow/deny リストおよびエージェント定義の tools フィールドの挙動を実機検証した。結果として、Claude Agent SDK 経由の環境では allow/deny リストによる権限制御も tools フィールドによるツール制限も機能していないことが判明した。公式ドキュメントの記述と実機検証結果を統合した運用ガイド `docs/subagent-permission-guide.md` を作成し、改善提案をまとめた。

### ルール化候補（.claude/rules/ や roles/ に反映できるパターン）

| # | 知見 | 対象ファイル・領域 | 詳細 |
|---|------|-----------------|------|
| 1 | エージェント定義の `tools` フィールドは意図の文書化であり、環境によっては強制力がない | `.claude/skills/*/agents/*.md` | SDK 環境では tools フィールドの制限が適用されない。rules/ にこの前提を記載し、tools フィールドを「制限」ではなく「推奨ツールセットの宣言」として位置づけるべき |
| 2 | deny リストは CLI 対話型環境のセーフガードであり、絶対的な安全策ではない | `.claude/settings.json`, `.claude/rules/` | deny リストのみに依存したセキュリティ設計を避ける旨を rules/ に記載すべき |

### 参考情報（文脈依存の気づき・今後の参考）

| # | 知見 | 背景・文脈 |
|---|------|-----------|
| 1 | Permission 評価順序（Hooks → Deny → Permission mode → Allow → canUseTool）は環境によって適用の有無が異なる | CLI 環境では全ステップが機能するが、SDK 環境では Deny/Allow のステップがバイパスされる可能性がある |

## 所感・次フェーズへの申し送り

### 改善提案（T-005）

#### 提案1: settings.json の allow リストへの追加・削除

**提案**: 現状維持。追加・削除は不要。
**理由**: 現在の allow リストは開発プロセスに必要なコマンドを適切にカバーしている。`date`, `pwd` 等の非 allow コマンドは CLI 環境で対話的に確認すれば十分であり、allow リストに追加する必要性は低い。
**変更しない場合のリスク**: 低。CLI 環境で非 allow コマンド実行時に許可プロンプトが表示されるが、頻度は低い。

#### 提案2: エージェント定義への `permissionMode` / `disallowedTools` 明示記述

**提案**: 現時点では不要。`tools` フィールドのみで十分。
**理由**: `tools` フィールドが allowlist として機能する（CLI 環境前提）ため、重複して `disallowedTools` を設定する必要性は低い。`permissionMode` はプロジェクトレベルの設定であり、エージェント定義に記載するものではない。
**変更しない場合のリスク**: 低。tools フィールドが機能する CLI 環境では意図通りの制限が適用される。

#### 提案3: deny リストの強化

**提案**: `Bash(rm *)` パターンの追加を検討する。
**理由**: 現在の deny リストは `Bash(rm -rf *)` のみ。`rm -r`（`-f` なし）や `rm` 単体での削除は deny されない。意図しない削除を防ぐために `rm` コマンド全般を deny に含めることを推奨する。
**変更すべき理由**: `rm -rf` のみの deny では `rm -r` で回避可能。
**変更しない場合のリスク**: 中。`rm -r` による意図しない削除がセーフガードを通過する。ただし CLI 環境では allow リストにも `rm` が含まれていないため、許可プロンプトが表示される。

### その他申し送り

- 本検証は Claude Agent SDK 環境で実施したため、Claude Code CLI の対話型環境での挙動とは異なる可能性がある。CLI 環境での再検証が望ましい場合は別施策として起票を推奨する
- 00_proposal.md で報告された「allow リストに含まれるのに許可プロンプトが出る」現象については、本検証では再現しなかった。この現象は CLI 環境固有の可能性がある
