# 改善施策提案: スキル/プラグイン機能重複の定期チェック制度化

## 背景・課題

現在、`.claude/skills/` 配下に 25 件のスキルが存在し、`plugin-reference` スキルで 76 件のプラグインを評価済み（うち高適用性 4 件、中適用性 6 件）、`builtin-skills-reference` スキルでビルトインスキル 5 件を評価済みである。

しかし、これらの間で機能が重複していないかを体系的にチェックする仕組みがない。スキルの追加・変更が進むにつれ、以下のリスクが顕在化する:

1. **スキル間の機能重複**: 類似機能を持つスキルが独立に存在し、メンテナンスコストが増大する
2. **プラグインとスキルの重複**: 導入検討中のプラグインが既存スキルと同等の機能を提供している場合、導入判断が不正確になる
3. **ビルトインスキルとカスタムスキルの重複**: Claude Code のビルトインスキル（/simplify, /batch 等）とカスタムスキルの守備範囲が曖昧

## 目標

config-optimizer-manager の CO ターゲットに CO-008（スキル/プラグイン機能重複チェック）を追加し、定期的な config 最適化セッションの一環として重複チェックが自動的に実施される状態を実現する。

## スコープ
### やること
- config-optimizer-manager SKILL.md に CO-008 ターゲット定義を追加
- 3つのワーカーエージェント（collection, analysis, proposal）に CO-008 の手順を追加
- 評価者エージェントの CO ターゲット数を 7→8 に更新
- リファレンスファイルに CO-008 セクションを追加

### やらないこと
- 新規の独立スキル/セッションの作成（既存の config-optimizer に組み込む）
- 実際の重複解消作業（チェック制度の構築のみ）
- プラグインの実導入

## 期待される効果

- config 最適化セッション実行時に、スキル/プラグイン間の機能重複が自動的に検出される
- 重複検出結果が OPT-XXX 提案として backlog に起票され、統合・整理が促進される
- 設定の冗長性が段階的に排除され、メンテナンスコストが削減される

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| CO ターゲット追加により各ワーカーの処理時間が増加 | 低 | CO-008 は既存スキル/プラグインの description 比較が中心で、外部参照の負荷は小さい |
| 4つのエージェント定義への変更で整合性が崩れる | 中 | テーブル形式の追加で変更を最小化。評価者のカバレッジ基準も併せて更新 |

## 壁打ちの背景

壁打ちフェーズで以下を確認・決定した:

1. **組み込み先の選定**: config-optimizer（推奨）、triage、両方に分散配置の 3 案を検討。config-optimizer が `.claude/` 配下の設定最適化を本務としており、スキル/プラグイン重複チェックは自然な拡張であるため、config-optimizer を選択
2. **チェック形態**: 専用スキル新設ではなく、既存の定期セッション（config-optimizer）への CO ターゲット追加とした。これにより新規セッション管理のオーバーヘッドを回避
3. **現状調査**: プラグインは未導入（評価のみ）、スキル 25 件。重複検出の仕組みは分散的（TG-002, BM-005 等）だが、スキル/プラグイン間の機能重複に特化したチェックは存在しない

## 備考・設計パターン

CO-001〜CO-007 と同じ構造パターン（3フェーズ × ワーカー + 評価者）に従い、CO-008 を追加する。ハードコードされた列挙型の構造のため、4 つのエージェント定義すべてに手順セクションを追加する必要がある。
