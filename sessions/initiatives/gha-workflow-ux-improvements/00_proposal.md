# 改善施策提案: gha-workflow-ux-improvements

## 背景・課題

GHA ワークフロー経由の施策運用において、以下の UX 課題が蓄積している:

1. **backlog-to-issue のスケジュール間隔が短すぎる**: 30分間隔で実行されるが、バックログ候補の追加頻度に対して過剰。GitHub Actions の実行コストが無駄になっている
2. **`@ai-task` コマンドプレフィックスが冗長**: 毎回 `@ai-task` と入力する必要があり、操作効率が低い
3. **Issue 上のコマンド表示がコピーしにくい**: インラインコード（`` ` ``）でコマンドを表示しているが、GitHub のフェンスドコードブロックのコピーボタンが使えない
4. **feedback が一部コマンドで下流に渡らない**: `/start` 時の初期コンテキスト補足や、gate-review の `/approve` 時のフィードバックが下流ワークフローに伝播しない
5. **README.md が実態と乖離**: 存在しないワークフロー（`backlog-auto-execute.yml`, `backlog-candidate-propose.yml`）が記載され、存在するワークフロー（`initiative-batch-approve.yml`）が記載されていない

## 目標

- GHA ワークフローの操作性・保守性を改善し、Issue ベースの施策運用をよりスムーズにする
- 全コマンドで feedback を下流ワークフローに渡せるようにし、人間のフィードバックが確実に AI に届く設計にする

## スコープ
### やること
- backlog-to-issue.yml のスケジュール cron を `*/30 * * * *` → `0 */3 * * *` に変更
- initiative-dispatcher.yml のトリガー条件・コマンドパースを `@ai-task` → `@ai` に変更
- Issue コメント投稿箇所のコマンド表示をフェンスドコードブロックに変更
- initiative-close.yml に `feedback` input を追加し、dispatcher から渡す
- initiative-wallbash.yml の `/start` ディスパッチ時に args（feedback）を渡す
- gha-execute/SKILL.md, gha-close/SKILL.md, gha-wallbash/SKILL.md の prompt に feedback 参照を追加
- README.md を実態に合わせて全面更新（`@ai` への変更も反映）

### やらないこと
- ワークフローのロジック変更（ゲート判定ロジック、auto-dispatch 等）
- 新規ワークフローの追加
- GHA スキルの作業フロー自体の変更

## 期待される効果

- backlog-to-issue の実行回数が 1/6 に削減（48回/日 → 8回/日）
- コマンド入力が `@ai-task` → `@ai` で 4文字短縮
- フェンスドコードブロックにより GitHub UI のコピーボタンで1クリックコピー可能
- 全コマンドで人間のフィードバックが AI に確実に到達

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| `@ai` が他の bot やユーザーメンションと衝突 | 低 | `@ai` はユーザー名として使えないため GitHub メンションと衝突しない。他の bot が `@ai` を使う可能性は極めて低い |
| コマンド表示変更で既存 Issue のガイドが古くなる | 低 | 進行中の施策はメタデータコメントに旧コマンドが残るが、dispatcher 側で両方サポートする移行期間は不要（`@ai` のみに切り替え） |

## 壁打ちの背景

- `@ai-task` の完全撤廃ではなく `@ai` への短縮を選択。プレフィックスを維持することで誤発火防止・bot ループ防止・可読性を確保しつつ入力効率を改善する
- feedback の伝播は `/start` と `/approve`（gate-review）の2箇所がギャップ。他のコマンドは既に対応済み
- コマンド表示のコピーしやすさは、GitHub がフェンスドコードブロックにコピーボタンを自動付与する機能を活用する
