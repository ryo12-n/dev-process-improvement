# 改善施策提案: impl-worker-worklog-reliability

## 背景・課題

impl-worker（Phase 4 実装ワーカー）は `41_impl_work_log_W<N>.md` にチェックポイント（START / TEST-RUN / COMPLETE / ESCALATE）を手動記録する設計だが、以下の構造的要因により記録漏れが発生しやすい:

1. **チェックポイント記録がアクションから分離**: ワークフローで「実装する」と「記録する」が別ステップ
2. **テンプレートファイルが存在しない**: `07_issues_W_template.md` はあるが `41_impl_work_log_W_template.md` がない。マネージャーは空ファイルを作成するのみ
3. **フォーマットがエージェント定義にしか存在しない**: ワーカーはエージェント定義のフォーマット定義を参照しながらファイルに書く必要がある（認知負荷が高い）

## 目標

- impl-worker の作業ログ（チェックポイント）記録の構造的信頼性を向上させる
- 「記録を忘れる」から「セクションを埋めるだけ」への認知負荷の転換

## スコープ
### やること
- 作業ログテンプレート `41_impl_work_log_W_template.md` の新規作成
- `impl-worker.md` のワークフロー再構成（チェックポイント記録をアクションに融合）
- `l1-impl-manager/SKILL.md` の事前準備・事後検証セクション強化
- `docs/workflow.md` の連動更新

### やらないこと
- PostToolUse Hook による自動記録（サブエージェントでの Hook 発火が不確実）
- 外部 API 依存の仕組み導入
- `manager-common-policy` への変更（impl-worker 固有の改善）

## 期待される効果

- チェックポイント記録率の向上（テンプレートによる構造的ガイド）
- マネージャー事後検証での復元作業の削減
- ワーカーの認知負荷低減

## リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| テンプレートが冗長すぎてワーカーの負担が増える | 中 | 最小限のスケルトンに留める |
| impl-worker.md の変更が既存フローと整合しない | 中 | 関連ファイル一覧を確認し連動更新する |

## 壁打ちの背景

inbox の Claude Context Manager（PostToolUse Hook で diff を自動要約）アプローチを評価した結果:
- 構造化チェックポイント（特定フィールド）vs free-form 要約で不一致
- サブエージェント実行での Hook 発火が不確実
- 外部 API 依存が制約違反

ただし「テンプレートで構造を事前提示し、埋めるだけにする」設計思想は有用と判断。この思想を4戦略に展開する。
