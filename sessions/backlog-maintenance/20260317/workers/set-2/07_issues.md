# 課題メモ（ワーカーセット別バッファ）: Set-2

> **このファイルの役割**: ワーカーセット内で発見された課題・知見・リスクの一時メモ置き場。
> マネージャーが `03_report.md` 作成時に全ワーカーセットの課題を集約し、`プロセス改善_課題管理.csv` へ転記する。

## 転記ルール

- ワーカー・評価者は課題をこのファイルに起票する（CSV 直接起票はしない）
- マネージャーが集約時に CSV 転記の要否を判断する
- 施策スコープ外・他のセッションでも再発しうる課題 → **マネージャーが CSV に転記**
- 当セッション内だけで完結する軽微な修正メモ → 転記不要（クローズ扱い）
- 転記済みの項目には `[転記済 ISS-XXX]` を末尾に追記する

### [2026-03-17] BL ID 二重採番による BM-005 走査効率低下
- **種別**: 改善提案
- **発見者**: ワーカー
- **内容**: BL-076 と BL-077 がそれぞれ2件ずつ存在する（二重採番）。BM-005 スキャンレポート内で区別記載が必要になり、「BL-076(aidd設計乖離)」「BL-076(aiddテストインフラ)」のような冗長な表記を使わざるを得なかった。backlog.csv 上でも2行が同一 ID で存在しており、CSV ベースの機械的集計に支障がある。
- **対応方針（案）**: BL-084（並列セッション環境での BL ID 衝突防止）の優先対応を推奨。暫定対応として、二重採番された4件のうち後発2件に新しい BL-ID を再採番することを検討。
- **関連タスク**: BM-005

### [2026-03-17] 新規エントリの記述粒度不足が重複検出精度に影響
- **種別**: 改善提案
- **発見者**: ワーカー
- **内容**: BL-084〜BL-093（直近起票分）のエントリは「課題・背景」が1-2文の簡潔な記述のみで、「関連エントリ」や「補足・参考情報」が空欄のものが多い。差分ベーススキャン（BM-005）では各エントリの記述内容を比較材料にするため、記述粒度が低いと重複判定の精度が下がる。
- **対応方針（案）**: backlog エントリ起票時の最低記述基準を backlog テンプレートまたは起票チェックリストに追加する（例: 「課題・背景」は3文以上、関連エントリがあれば記載必須）。BL-077(旧フォーマット統一) の施策スコープに含めることも検討。
- **関連タスク**: BM-005
